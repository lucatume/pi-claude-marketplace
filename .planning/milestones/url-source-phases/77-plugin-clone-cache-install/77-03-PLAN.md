<objective>
Build the plugin clone-cache seam that clones a git plugin source at its pinned/resolved sha into the shared source-addressed cache (`plugin-clones/<key>/`), deduped by url+sha (PURL-04), with warm-cache reuse that stays offline (PURL-02). Add the supporting `resolveRemoteRef` primitive so an unpinned source can resolve remote HEAD to a sha WITHOUT a full clone (D-77-05).

Purpose: install.ts is forbidden the git surface by the `no-orchestrator-network` architecture gate. The clone lives in this sibling seam that install.ts calls by name. The seam mirrors `addGitClonedInGuard` (staging clone -> atomic rename -> MA-9 cleanup), adds a warm-cache short-circuit (dedup + offline), pins the exact commit (sha over ref), and tolerates the concurrent-install rename race.
Output: new `orchestrators/plugin/clone-cache.ts`, a `resolveRemoteRef` wrapper in `platform/git.ts`, a new `resolveRemoteRef` method on the `GitOps` interface + `DEFAULT_GIT_OPS`, and `tests/orchestrators/plugin/clone-cache.test.ts`.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| remote git host -> staging clone | An untrusted repo is cloned into staging before validation; must land atomically or clean up (NFR-1/NFR-3) |
| concurrent installs -> shared cache dir | Two installs of the same url+sha race the rename into `plugin-clones/<key>/` |
| clone error -> user output | A clone failure error must not leak credentials (none handled this phase — public-only) |

Tests assert cleanup-on-throw. |
