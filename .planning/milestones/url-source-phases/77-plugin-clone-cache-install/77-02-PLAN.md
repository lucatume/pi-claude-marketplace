<objective>
Widen the resolver so `url` / `git-subdir` / `github`-object plugin sources classify installable (PURL-01), while keeping the resolver strictly network-free by injecting the clone-vs-probe policy as an optional `ResolveContext` callback. For git sources, derive the plugin root and its NFR-10 containment from the injected callback's result, anchored to the plugin's OWN clone root (PURL-03), never `marketplaceRoot`.

Purpose: The resolver is called by BOTH `install` (may clone) AND the network-free `list`/`info` surfaces. The clone must NOT happen in the resolver. Injecting `resolveGitPluginRoot` lets install supply a clone-materializing callback while list/info supply nothing (git sources then resolve unavailable â€” the correct path-only back-compat) or a cache-presence-only probe. This preserves the `no-orchestrator-network` architecture gate on list.ts without change.
Output: extended `domain/resolver.ts` (widened `sourceUnsupportedReason`, new `GitPluginRootResult` union, `resolveGitPluginRoot` on `ResolveContext`, branched `preflightStages` git-source derivation) and extended `tests/domain/resolver-strict.test.ts`.
</objective>
