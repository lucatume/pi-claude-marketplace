<objective>
Widen the three marketplace lifecycle orchestrators (`add`, `update`, `remove`)
to handle `url`-kind sources by mirroring their existing `github` arms with
exactly two differences: the clone URL is `source.url` verbatim (D-76-06), and
NO auth bundle is constructed (D-76-07, public-only). Add the `authentication
required` classification arm to `classifyAddError` so a 401/403 clone challenge
surfaces truthfully. This delivers MURL-01 (add), MURL-03 (update), and MURL-04
(remove) end-to-end for url marketplaces.

Purpose: These are the request-response + file-I/O seams where the clone URL and
auth policy are chosen. Every downstream step (manifest validation, duplicate
check, stale-clone check, atomic rename, state record, config write-back,
clone deletion) is already source-kind-uniform once the URL is chosen — the work
is gate-widening, not re-implementation.

Output: An `add.ts` with a shared clone-into-guard helper and a url path, an
`update.ts` with a url refresh arm, a `remove.ts` with a widened clone-deletion
gate, a `platform/git.ts` clone doc contract that no longer claims github-only,
and extended orchestrator tests.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| orchestrator → isomorphic-git clone/fetch | The chosen clone URL (now any `https://` host) crosses into the git primitive; network reached on `add`/`update` only |
| clone error → user notification | An HTTP error from a private/nonexistent repo crosses back into the reason-classification + notify surface |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-76-04 | Information disclosure | `classifyAddError` HttpError arm | low | mitigate | The `authentication required` reason carries only the HTTP status (401/403), never a token — Phase 76 constructs no auth for url sources (D-76-07). No token can appear in the cause chain because none exists. |
| T-76-05 | Elevation of privilege | url path constructing github auth "to be safe" | medium | mitigate | D-76-07 forbids ANY auth bundle for url sources. The url path passes no `auth`; the clone-options `auth` key is absent (test-asserted). This prevents leaking github credentials to a non-github host. |
| T-76-06 | Denial of service | orphaned url clone permanently blocking re-add | medium | mitigate | remove.ts widens the clone-deletion gate to `github || url` so a removed url marketplace leaves no `sources/<name>/` orphan (NFR-3). Test asserts re-add after remove succeeds without `{stale clone}`. |
| T-76-07 | Information disclosure (SSRF-ish) | clone of an internal `https://` host | low | accept | Out of Phase 76 scope to block (public-repo utility requires arbitrary hosts). `https://`-only (D-76-01, enforced by the Plan-01 parser) blocks `file://`/`ssh://` local-resource reach; no pre-clone probe (fail-clean is the signal, NFR-3). |
| T-76-SC | Tampering | npm/pip/cargo installs | n/a | accept | This plan installs no packages; `isomorphic-git` is already a committed direct dependency. No legitimacy checkpoint required. |
</threat_model>
