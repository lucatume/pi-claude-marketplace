<objective>
Make `reinstall` re-materialize a git-source plugin from the state record's recorded resolvedSha
(D-78-02) so PURL-07's offline guarantee holds unconditionally on a warm cache. Reinstall must NOT
call resolvePluginPin (which resolves unpinned HEAD via a network resolveRemoteRef) — the recorded
sha IS the pin. It reaches materializePluginClone BY NAME through a ReinstallCloneCacheSeam bundle
(mirroring install's InstallCloneCacheSeam) so reinstall.ts never names gitOps and stays in the
no-orchestrator-network forbidden list, gate-clean.

Two edits:
1. Inject a recorded-sha probe at resolveInstallable (reinstall.ts:1268-1275, currently no git callback
   -> git plugins resolve unavailable). The probe: pin = oldRecord.resolvedSha, cloneUrl =
   canonicalCloneUrl(source) (Plan 02, pure, no network), then materializePluginClone(url, pin) directly;
   git-subdir uses resolveGitSubdirRoot (Plan 02). No resolvePluginPin, no resolveRemoteRef.
2. Carry oldRecord.resolvedSha forward in updateStateRecord (reinstall.ts:1437) — it currently DROPS the
   field, corrupting GC key derivation and a later reinstall's pin.

Output: seam bundle + probe + resolvedSha carry-forward in reinstall.ts; extended reinstall.test.ts.
</objective>
