<objective>
Make `list` and `info` render git-source plugins with correct status while staying strictly
network-free (NFR-5). Two behaviors:

1. Uninstalled git plugin renders (available), not (unavailable) — D-78-03, the load-bearing
   decision. The resolver maps a not-cached probe result to unavailable {not installed}
   (resolver.ts:676-679), AND a materialized result requires the pluginRoot to exist on disk
   (preflightStages line 737 statKindOf !== "dir" -> unavailable) — so no sentinel pluginRoot can
   force installable for an entry with no clone. Therefore a not-installed git entry must be
   classified (available) directly from the manifest, WITHOUT invoking the resolver's git arm,
   exactly like a not-installed path entry (whose marketplace clone DOES exist and validates). The
   existing isLocallyResolvable(parsedSource) gate in info.ts (def at 107; used at 901, 1056) is the seam.

2. Installed git plugin with a missing clone shows no status change — D-78-04, satisfied by
   construction. list's installedRowMessage derives status from recorded compatibility/version and
   calls the resolver ONLY when upgradable; info's buildInstalledRow short-circuits non-path sources
   to buildNonPathInstalledRow (info.ts:901-903) which reads the recorded record. Neither probes the
   clone dir for installed status, so a missing clone is invisible. Add the presence-only probe at the
   upgradable-candidate resolve so a git upgradable candidate degrades to plain (upgradable) rather
   than (unavailable), but add NO clone-dir probing to the installed status path.

Output: a shared fs-only presence probe + a not-installed git (available) short-circuit at the
list/info row builders; list.test.ts + info.test.ts extended.
</objective>
