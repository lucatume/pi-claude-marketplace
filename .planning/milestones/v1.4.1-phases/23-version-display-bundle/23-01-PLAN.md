| filesystem -> resolvePluginVersion | `<installable.pluginRoot>/.claude-plugin/plugin.json` is read from disk inside a validated installable's plugin root; its bytes are untrusted input (could be absent, truncated, malformed, or contain a non-string `version`). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-23-01 | Tampering | `resolvePluginVersion` path resolution | mitigate | Read path is built only from `installable.pluginRoot` (already containment-validated by the resolver per NFR-10) joined to the fixed literals `".claude-plugin"` + `"plugin.json"` via `path.join`; no user/network-controlled path segment is interpolated, so the read cannot escape the resolved plugin root. |
| T-23-02 | Denial of Service | `JSON.parse` of plugin.json | mitigate | Read+parse is wrapped in a single `try/catch`; ENOENT, malformed JSON, missing `.version`, and non-string `.version` all fall through to the next tier (D-23-02) -- the function never throws, so a hostile/garbage plugin.json cannot break install/update/reinstall. |
| T-23-03 | Information disclosure | recorded `state.version` | accept | The version string is already surfaced via `notify()` and persisted today; promoting plugin.json's declared version to tier-1 exposes only a value the plugin author chose to publish. No new secret crosses a boundary. |
| T-23-SC | Tampering | npm/pip/cargo installs | accept | No package-manager installs in this plan; only the built-in `node:fs/promises` `readFile` is newly imported (already a project convention per `version.ts:21`). No `## Package Legitimacy Audit` required. |
