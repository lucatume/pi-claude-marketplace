## Cross-Phase Integration (4/4 verified)

The 5 phases land on intersecting code paths in the transaction layer:

1. **Phase-ledger ↔ bridge commit** (Phase 37 ↔ Phase 38): `runPhases` invokes
   `phase.do(ctx)` which calls per-bridge `commitPrepared*`. Phase 37's failing-phase
   undo runs BEFORE Phase 38's reverse-walk over executed[], so a partial bridge
   commit triggers (1) bridge-internal completedRenames reverse-walk THEN
   (2) phase-ledger compensation. Integration verified by Phase 38's
   `tests/bridges/agents/stage.test.ts` rollback tests still passing under Phase 37's
   new undo-call ordering.

2. **Bridge commit ↔ cascade** (Phase 38 ↔ Phase 39): Phase 39's cascade-failure
   filter on uninstall.ts uses `outcome.dropped.*` populated by `cascadeUnstagePlugin`,
   which itself drives the bridge commit-paths Phase 38 hardened. Field-name
   asymmetry (`dropped.commands` → `resources.prompts`) is verified consistent
   across uninstall.ts and remove.ts (grep at both sites).

3. **Cascade ↔ update reorder** (Phase 39 ↔ Phase 40): Phase 40's `finalizeUpdateRecord`
   uses the same per-bridge gating pattern (`!failedPhases.has(bridge)`) and the same
   field-name mapping as Phase 39's filter. Both phases lock the
   commands→prompts asymmetry at their respective fix sites.

4. **Phase-ledger compensation ↔ update intent-mark** (Phase 37 ↔ Phase 40):
   Phase 40's two-`withStateGuard` discipline (intent-mark before commits, finalize
   after) is preserved under Phase 37's catch-block restructure -- the PUP-6 phase-3
   failure test at `tests/orchestrators/plugin/update.test.ts:744` (which exercises
   the full pipeline) stays GREEN, byte-unchanged for the test body.
