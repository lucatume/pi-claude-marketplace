Add `notifications: { mode: "standalone" | "orchestrated" }` to the four driven orchestrators that Phase 55 Plan 02's `applyReconcile` will compose — mirroring the `InstallPluginNotifications` precedent already established by `installPlugin` in Phase 19. Purpose: This is a foundation plan. `applyReconcile` (Plan 02) must aggregate per-entry outcomes from N orchestrator calls into ONE `notify()` per load (IL-2). Without a uniform orchestrated mode, apply.ts would have to inject a swallow-and-capture ctx OR per-orchestrator try/catch the throws — both fragile (RESEARCH Pattern 4 Strategy B; the chief code-design tension of Phase 55 per Research Open Question 1).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller -> orchestrator | The new `notifications` option is caller-supplied; an untrusted caller could request orchestrated mode to suppress error notifications and conceal failures from the user. |
| orchestrator -> error classifier | Catastrophic-failure fallback classification surfaces an Error message in `cause`; raw error messages can leak filesystem paths or auth tokens. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-55-01-01 | Information disclosure | Orchestrated-mode `cause` / `error` fields returned to caller | mitigate | The orchestrated outcome is consumed by `applyReconcile` (Plan 02), not an end user. apply.ts will project `outcome.reason` (closed-set ContentReason) into the cascade and NEVER render the raw `error.message` in user-visible output. Document this contract in apply.ts (Plan 02) — Plan 01 only exposes the typed surface. |
| T-55-01-02 | Tampering | Hostile caller passes `notifications: { mode: "orchestrated" }` to suppress error visibility | accept | Risk is bounded: the four orchestrators are package-internal exports; no public IPC / extension API exposes them. The trust boundary is the in-process call graph, not an external surface. |
| T-55-01-03 | Repudiation | Failures in orchestrated mode produce no log unless caller renders them | mitigate | The caller (Plan 02 apply.ts) is the sole consumer and is contractually required to emit a single notify() that includes every per-entry failure (IL-2 + RECON-03 + RECON-04). Standalone mode default ensures no existing caller silently drops error notifications. |
| T-55-01-SC | Tampering | npm/pip/cargo installs | mitigate | No new package installs in this plan. Package legitimacy gate is N/A per 55-RESEARCH.md Package Legitimacy Audit ("Skipped — no external packages added in Phase 55"). |
</threat_model>
