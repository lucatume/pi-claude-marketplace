<objective>
Phase 52 ships the first-run migration safety rail: when a user upgrades into v1.12 with an existing populated `state.json` but no `claude-plugins.json`, the extension generates a lossless `claude-plugins.json` from state on the next load so the Phase 55 reconcile (when it lands) never sees "no config -> empty desired state" and prunes everything.

This plan delivers the entire phase: the new `persistence/migrate-config.ts` module (pure `buildConfigFromState` projection + thin ENOENT-gated `migrateFirstRunConfig` orchestrator), a populated `tests/persistence/fixtures/legacy/state-populated-mixed.json` fixture exercising every MIG-01/MIG-02 anchor, and the `tests/persistence/migrate-config.test.ts` suite (Wave-0 RED-first) that locks correctness.

Purpose: Close MIG-01 (lossless first-run generation) and MIG-02 (atomic, idempotent, convergence-ready) at the data and seam levels. The load-wiring caller (who invokes `migrateFirstRunConfig` and in what order relative to reconcile) is deferred to Phase 55; the planner-level `planReconcile` no-op exit gate is deferred to Phase 53 (where `planReconcile` will exist) — this plan satisfies Success Criterion 4 at the data level here and explicitly marks the planner-level proof as a Phase 53 obligation.

Output: 1 new production file, 1 new test file, 1 new fixture file. Zero changes to `config-io.ts`, `state-io.ts`, `migrate.ts`, `config-merge.ts`, `locations.ts`, or the SPLIT-02 architecture test.

## Phase Goal (sourced from ROADMAP Phase 52)

A Pi user upgrading into v1.12 with an existing install gets a `claude-plugins.json` generated losslessly from their current `state.json` on first load, with nothing uninstalled — the safety rail that guarantees an existing install is never reconciled against absence.
