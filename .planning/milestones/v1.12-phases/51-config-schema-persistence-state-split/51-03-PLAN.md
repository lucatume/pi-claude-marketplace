<objective>
Close SPLIT-02: the structural enforcement that machine records are written ONLY to the
internal state file and the user config is written ONLY by command write-back / migration
— so no reconcile-path code (Phases 53-55) can silently grow a code path that writes the
user config, and no orchestrator can grow a code path that writes the state file outside
the sanctioned seams.

Purpose: Plans 01 and 02 introduced the persistence-layer seams. This plan locks ownership
at the architecture level so the seams cannot be bypassed as v1.12's downstream phases
(52-56) and any future milestone add new code paths. The test mirrors the existing
`tests/architecture/no-shell-out.test.ts` shape (walker + allow-list + 'exactly N' sibling
assertion) — the same shape that already locks `no-shell-out`, `no-telemetry-deps`,
`import-boundaries`, and other structural invariants.

Output: `tests/architecture/config-state-write-seams.test.ts` (NEW). No production code is
modified in this plan.

Per D-XX traceability: implements the SPLIT-02 architecture-test deliverable explicitly
called out in CONTEXT.md "Claude's Discretion" (item: "How the architecture test enforces
the SPLIT-02 write-seam ownership"). The discretion was: import-graph vs. call-site
assertion style. This plan chooses the call-site regex-on-source style (matching the
existing `no-shell-out.test.ts` template) because the write seam is a small, named set of
call patterns (`atomicWriteJson(loc.X, ...)`) and a regex walker is both the simplest and
the closest in-tree analog (PATTERNS.md: "exact" match).
</objective>
