<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User-authored config files → `loadConfig` | A hand-edited or VCS-checked-in `claude-plugins.json` enters the extension across this boundary. Per CFG-03, malformed/truncated input must abort, never silently coerce to empty desired state. |
| In-memory `ScopeConfig` → `saveConfig` → disk | `saveConfig` may be called with a `filePath` arbitrarily derived inside the extension (Phase 56 will do this). Per NFR-10 + SPLIT-02, every write site must containment-check before writing. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-51-01-01 | Tampering / Denial-of-Service | `loadConfig` reading a 0-byte / truncated / malformed `claude-plugins.json` | mitigate | Discriminated `ConfigLoadResult` (D-15): 0-byte and JSON-parse failures land in `{ status: "invalid", ... }`, never `valid` with empty config. No try/catch+default anywhere in `config-io.ts`. The Pitfall 51-1 anchor test (`loadConfig` on 0-byte file → `invalid`) locks this at unit-test time. Downstream Phase 55 reconcile aborts on `invalid` per CFG-03 — Phase 51 only locks the seam shape so the abort is structurally possible. |
| T-51-01-02 | Tampering | `loadConfig` on a JSON-valid but schema-invalid file (e.g., `marketplaces: "not an object"`, `schemaVersion: 2`) | mitigate | `CONFIG_VALIDATOR.Check(parsed)` failure returns `{ status: "invalid", error: "schema validation failed: <path>: <message>" }`. `firstConfigValidationErrorDetail` names the offending field for actionable error output. Schema-strict on KNOWN keys (D-09 lenient is only for UNKNOWN keys). |
| T-51-01-03 | Tampering | `saveConfig` called with a `filePath` that escapes `scopeRoot` (containment bypass) | mitigate | `await assertPathInside(scopeRoot, filePath, "saveConfig")` runs BEFORE `atomicWriteJson`. The `PathContainmentError` is thrown by the shared seam; `saveConfig` does NOT catch it. The architecture test in Plan 03 will additionally pin write-seam ownership to `saveConfig` exclusively. The unit test in this plan covers the runtime refusal. |
| T-51-01-04 | Tampering | Symlink at `<scopeRoot>/claude-plugins.json` or `<scopeRoot>/claude-plugins.local.json` pointing outside `scopeRoot` | mitigate | `assertPathInside` walks every parent segment with `lstat` and refuses any symlink in the chain (D-14 in `path-safety.ts`). The leaf-file symlink case is also rejected. Inherited from existing seam — no new mitigation required, but documented for SPLIT-02 closure. |
| T-51-01-05 | Tampering / DoS | Power loss mid-write of `claude-plugins.json` | mitigate | `saveConfig` writes through `atomicWriteJson` which uses `write-file-atomic` (tmp + fsync + rename + parent-dir fsync). NFR-1 satisfied. No new code path. |
| T-51-01-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages introduced in Phase 51-01 (verified in RESEARCH "Standard Stack": `typebox`, `write-file-atomic` are already in `package.json` from shipped milestones v1.0-v1.11). Carry-forward only; the milestone-level Package Legitimacy Audit in `51-RESEARCH.md` covers the existing packages. No blocking-human checkpoint needed. |
