<objective>
Close TR-01, TR-05, and TR-06 in a single plan covering all three bridges with one
new helper. Phase 38 hardens the agents and commands bridge commit paths to be
atomic at rename granularity (TR-01, TR-05) and replaces the brittle
`if (pathExists(pair.to)) throw` guard in all three `replacePrepared*` helpers
with a shared `removeOrphanIfPresent` primitive that pre-removes owned orphan
content while preserving the PI-6 foreign-content guard (TR-06).

Purpose: Today `commitPreparedAgents` step-2 fires `Promise.all` over N renames;
if rename M throws (e.g., EACCES on a target whose parent lost +x), the M-1
successful renames leak as orphan files at the target dir while the original
staging dir is also gone -- the install is wedged in a state no retry can clean
up because the staged sources are missing AND the targets are partial. The same
shape applies to `commitPreparedCommands` (already sequential but with no
rollback tracking). Separately, the three `replacePrepared*` helpers refuse to
rename when ANY content exists at `pair.to` -- even when state.json proves the
content is the install's own orphan from a prior partial run, which blocks
reinstall. The fix introduces ONE kind-strict helper (TR-06) that pre-removes a
target only when the caller has already confirmed ownership by basename
membership in the prepared handle's `_previousNames` / `_previousEntries` view,
and rewrites the two commit-path step-2 loops to sequential-with-rollback shape
borrowed from the existing `rollbackReplacementCommon` (spread-before-reverse,
leaks accumulation, never-throws-from-rollback). The PI-6 cross-plugin guard
is preserved by the call-site basename discriminator, NOT by helper internals.

Output: One shared helper added (~25-line export with JSDoc), two
commit-path step-2 loops restructured (~20 lines each), three `replacePrepared*`
rename loops adopt a 3-arm policy (~5 lines each call site), 5 unit tests for
the helper, 4 regression tests across the agents + commands bridges, 1 orphan-
tolerance test for skills. No new dependencies. No new files. Per RESEARCH
"Don't Hand-Roll" verdict: no `commitWithRollback` abstraction (Open Question
shape mismatch), no `MultiLeakError` (existing `appendLeaks` is sufficient),
no internalized ownership in the helper (Open Question 3 resolution).
</objective>
