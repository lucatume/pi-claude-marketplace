<decisions>
## Implementation Decisions

### Direction (CR-01 disposition)
- **D-17.2-01:** Fix the renderer to honor the documented orphan-fold contract (REVIEW recommendation, option a). Rejected: rewriting the docs to match the current "always emit when scope is defined" impl (option b) -- that would weaken the user-contract and force Phase 18+ orchestrators to omit `p.scope` manually on same-scope rows. Rejected: the hybrid forgiving signature (option c) -- it leaves the contract ambiguous and complicates the orphan-fold semantics.
- **D-17.2-02:** New signature: `renderScopeBracket(pluginScope: Scope | undefined, mpScope: Scope): string`. Return `""` when `pluginScope === undefined || pluginScope === mpScope`; otherwise return `` `[${pluginScope}]` ``. `mpScope` is non-optional -- the renderer always has the parent marketplace's scope from `composeMarketplaceBlock` and there is no reason to forge a fallback.
- **D-17.2-03:** Threading: `composePluginLines` and `renderPluginRow` both take an additional `mpScope: Scope` parameter. The threading is mechanical -- `composeMarketplaceBlock` already binds `mp.scope` via its local `mp` argument and just passes it down.

### Cleanup scope
- **D-17.2-04:** Fold ALL six 17.1 review warnings (WR-01..WR-06) into Phase 17.2 as a single notify.ts / catalog hygiene sweep. Rationale: WR-01..WR-06 share file proximity (4 of 6 are inside `shared/notify.ts` or directly testing it), share planning overhead (single phase, single set of plan files), and a follow-on "hygiene-only" phase would duplicate context-loading work. Phase blast radius widens past the literal "renderScopeBracket fix" goal but stays inside the notify.ts / catalog-uat / pi-api surface.
- **D-17.2-05:** Catalog `project-orphan-folded` byte form is updated (line 191 same-scope row drops `[user]`); fixture workaround at `tests/architecture/catalog-uat.test.ts:303-309` is removed and the misleading comment is rewritten or deleted. Other scope-bearing fixtures across both test files are audited; any same-scope row that currently includes an explicit `scope` matching the parent now produces no bracket -- fixture inputs can keep the explicit `scope` (harmless) or drop it (cleaner) at the planner's discretion, but byte-form expectations must reflect the corrected rule.
- **D-17.2-06:** WR-02 narrative claims at `docs/output-catalog.md:182` / `:196` are rewritten to match the corrected rule. The line-196 prose is already aspirationally correct; tightening wording is sufficient. No restructuring of the catalog beyond the affected paragraphs.

### Test coverage
- **D-17.2-07:** Add 3-4 focused byte-equality unit tests in `tests/shared/notify-v2.test.ts` covering: (a) same-scope `installed` row → no bracket, (b) orphan-fold `installed` row → bracket present, (c) same-scope `updated` row → no bracket, (d) orphan-fold `failed` row → bracket present. Reuse the existing mock-`ctx` pattern in that file. No new type-level proof in `notify-types.test.ts` (the signature change is reflected in the source itself; further compile-time locking is over-engineering for a one-off fix).
- **D-17.2-08:** WR-04 soft-dep probe tests live in `tests/platform/pi-api.test.ts` if that file exists (or are added there); otherwise extend `tests/shared/notify-v2.test.ts`. Planner picks based on existing file layout -- both placements are acceptable.

### Phase 18 hand-off
- **D-17.2-09:** No special prep for Phase 18+ orchestrator migration beyond fixing the renderer. Phase 18 builds normally on top of the corrected `renderScopeBracket` -- orchestrators construct `PluginNotificationMessage` payloads with `p.scope` from domain records, and the now-correct renderer suppresses the bracket on same-scope rows automatically. No "natural construction" contract test, no extra JSDoc.

### Recurrence guard

The new unit tests (D-17.2-07) lock the renderer-level rule for representative cases; the catalog UAT continues to lock the rendered surface across the full catalog. Adding a separate "renderer-vs-docs invariant" test is over-engineering for a one-time correction.

### Sequencing inside the phase

Suggested grouping: (Wave 1) CR-01 renderer fix + threading + new byte-equality unit tests; (Wave 1 or 2) catalog byte-form + fixture cleanup + WR-02 narrative; (Wave 2 or 3) WR-01 docstring refresh, WR-03 composeVersionArrow simplification, WR-04 soft-dep probe tests, WR-05 dead-helper deletion, WR-06 exhaustiveness gate hardening -- these are largely independent and can parallelize.
