<objective>
Close UAT gap G-21-01: `/claude:plugin list` emits a spurious `/reload to pick up changes` trailer when no state was changed (every list invocation with at least one installed plugin -- the common case). Root cause is status-token semantic overload: `PluginInstalledMessage` is used by BOTH state-transition cascade rows (install/update where reload-hint is correct) AND by `list.ts::installedRowMessage`'s steady-state inventory rows (where reload-hint is incorrect). `shouldEmitReloadHint` cannot distinguish the two senses.

Fix: introduce a new list-only status token `"present"` for `installedRowMessage` (mirrors the existing list-only `available` / `unavailable` / `upgradable` tokens). The four state-change tokens stay structurally unchanged and continue to drive the reload-hint correctly.

Purpose: SNM-15 requires reload-hint computation from message contents. The current implementation is correct in principle but semantically overloaded on a single token. This gap tightens the SNM-15 surface so `shouldEmitReloadHint`'s contents-derived decision is unambiguous: every status discriminator either always-triggers or never-triggers; no token straddles both surfaces.

Output: typed list-only inventory token + renderer arm + 2 new notify-v2 unit tests proving the bug is fixed + catalog-uat byte-equality re-passes after fixture + markdown updates. Single atomic commit per the established Phase 21 atomic-single-commit discipline (D-21-06).
</objective>
