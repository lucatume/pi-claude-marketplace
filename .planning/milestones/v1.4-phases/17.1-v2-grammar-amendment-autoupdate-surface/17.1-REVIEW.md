# Phase 17.1: Code Review Report

**Reviewed:** 2026-05-26
**Depth:** standard
**Files Reviewed:** 7
**Status:** issues_found

## Summary

Phase 17.1 amends the V2 grammar to expose three new `MarketplaceStatus`
literals (`autoupdate enabled`, `autoupdate disabled`, `skipped`), adds an
optional `reasons?` to `MarketplaceNotificationMessage`, extends
`renderMpHeader` with three new switch arms, refactors the severity ladder,
extends the reload-hint trigger ladder, and rewrites the
`marketplace autoupdate` catalog section with five per-state blocks. The
amendment lands cleanly across types, renderer, catalog, and tests, and the
arch-level closed-set membership proofs in `tests/architecture/notify-types.test.ts`
were updated to lock the new tuple length (7) and the expected literal set.

The implementation is largely consistent with the documented amendment, but
this review surfaces one **BLOCKER**: a real divergence between the documented
plugin-row scope-bracket contract and the actual `renderScopeBracket`
implementation. Several **WARNINGS** flag duplicated logic, defensive code
that masks real type contracts, an unused-helper preservation hack in the
fixture file, and one documentation/implementation drift the catalog
description hides via fixture workarounds. **INFO** findings cover stale
docstrings, minor dead branches, and one cleanup opportunity.

The Phase 17.1-specific additions themselves (the 3 new arms, the `reasons?`
field, the severity-ladder extension, the reload-hint extension) are
self-consistent and traceable to D-17.1-01..05.

## Critical Issues

### CR-01: `renderScopeBracket` does not implement the documented "orphan-fold only" contract

**File:** `extensions/pi-claude-marketplace/shared/notify.ts:683-685`
**Issue:** The renderer emits the plugin-row `[<scope>]` bracket whenever
`p.scope !== undefined`, but the binding contract documented in
`docs/messaging-style-guide.md:73` and `docs/output-catalog.md:39` requires
the bracket only when `p.scope !== undefined` **AND** `p.scope !== mp.scope`
(the orphan-fold case per D-16-17):

```ts
function renderScopeBracket(scope: Scope | undefined): string {
  return scope === undefined ? "" : `[${scope}]`;
}
```

The function receives only `p.scope`; it never compares against the parent
marketplace's scope. Consequence: any orchestrator that explicitly sets
`p.scope` to the *same* value as `mp.scope` (which is reasonable when the
domain object carries the scope as a plain field on every plugin) will emit
the bracket on a row where the spec says it should be omitted. The catalog
state `project-orphan-folded` (catalog-uat.test.ts:262-314) only works
because the fixture deliberately adds a comment-laden workaround in lines
303-309 explaining that the user-scope alpha row must also carry an explicit
`scope: "user"` to defeat the "same-scope rows omit the bracket" rule -- which
the renderer does not actually enforce. The catalog row at output-catalog.md:191
(`alpha [user] v1.0.0 (installed)`) is only byte-accurate against the renderer
because the fixture is constructed to match. A naturally-built orchestrator
payload will not match the documented contract.

This will surface as either (1) a user-visible drift the moment a Phase 18-20
orchestrator passes `p.scope` straight through from a domain record, or (2)
a catalog UAT failure when a fixture is constructed without the workaround.

**Fix:** Make `renderScopeBracket` take the parent scope and compare, then
update every call site to pass `mp.scope`:

```ts
function renderScopeBracket(pluginScope: Scope | undefined, mpScope: Scope): string {
  if (pluginScope === undefined || pluginScope === mpScope) {
    return "";
  }
  return `[${pluginScope}]`;
}
```

And thread `mp.scope` from `composeMarketplaceBlock` through `composePluginLines`
into `renderPluginRow`. Then the `project-orphan-folded` fixture can drop the
artificial second `scope: "user"` (lines 303-309) and the `same-plugin-both-scopes`
catalog narrative ("same-scope rows omit the bracket because `p.scope === mp.scope`")
will be true at runtime instead of a rationalisation.

Alternatively, if the *intent* is "explicit scope on the plugin row always
emits the bracket; orchestrators choose by setting/unsetting the field", then
the docs at messaging-style-guide.md:73 and output-catalog.md:39 must be
rewritten to match the implementation. The current state is the worst of both
worlds: docs and renderer disagree, and the catalog UAT passes only because a
fixture comment paves over the divergence.

## Warnings

### WR-01: Stale docstrings still describe Phase 1 V1-only `notify.ts` surface

**File:** `extensions/pi-claude-marketplace/shared/notify.ts:16-56, 97-105`
**Issue:** The top-of-file block comment (lines 16-56) describes the V1-only
contract ("Severity is part of the function name", "shared/notify.ts is the
SOLE sanctioned ctx.ui.notify call site", references to Phase 12 / Phase 13).
It does not mention the V2 `notify()` / V2 `notifyUsageError(ctx, message)`
entry points that this file has now exported for two phases. A code reader
landing on the file with no Phase context will think the V1 wrappers are the
public surface; in reality Phase 16 introduced the V2 entry points and Phase
17.1 finalised them. The block at lines 97-105 (above `notifyUsageError`)
similarly claims "Phase 6 will assemble actual Usage block strings ... Phase
1 ships the primitive only; call sites do not yet exist" -- both clauses are
now historical fiction.
**Fix:** Rewrite the top docstring to describe both V1 and V2 surfaces with
the Phase 21 deletion timeline, and update the `notifyUsageError` preamble to
acknowledge that V2 callers exist (catalog UAT, per-status tests) and the V1
3-arg path is deprecated awaiting Phase 21 deletion.

### WR-02: Catalog narrative claims justify byte forms with stale rules

**File:** `docs/output-catalog.md:182, 196`
**Issue:** Line 182 ("Plugin rows omit the scope bracket because
`p.scope === mp.scope`") and line 196 ("its row carries the explicit
`[project]` bracket because `plugin.scope !== marketplace.scope`") describe a
rule the renderer does not implement (see BLOCKER-1). The catalog states are
byte-accurate because the fixtures match the *implementation*, but the prose
narrative ascribes the byte forms to a rule (`p.scope === mp.scope`-keyed
omission) that does not exist in `renderScopeBracket`. Reviewers reading the
narrative are mis-led about what the catalog actually proves.
**Fix:** Pair with BLOCKER-1's fix. If the renderer is changed to honour the
documented rule, the prose is then accurate. If the docs are changed to match
the renderer ("the bracket is emitted whenever `p.scope` is defined"), the
prose narratives at lines 182 and 196 must be rewritten to match.

### WR-03: `composeVersionArrow` carries dead defensive branches the type model forbids

**File:** `extensions/pi-claude-marketplace/shared/notify.ts:701-719`
**Issue:** The `composeVersionArrow(from, to)` helper has four branches
covering `(undefined, undefined)`, `(defined, defined)`, `(undefined, defined)`,
and `(defined, undefined)`. The only live caller is the `updated` arm of
`renderPluginRow`, where Phase 15 D-15-04 makes both `from` and `to`
**REQUIRED** -- the type system guarantees `(defined, defined)` is the only
input shape that can statically reach this helper. The other three branches
are unreachable from any sanctioned caller. The docstring (lines 696-700)
admits this ("the only-`to` and only-`from` branches are defensive (no current
caller in this plan exercises them)"). Dead branches accumulate maintenance
cost; one of them (lines 710-712 for `to !== undefined`) inadvertently
`v`-prefixes a bare version which contradicts the surrounding "from is bare,
to is `v`-prefixed" convention -- a future caller would silently mis-render.
**Fix:** Replace the four-branch shape with the single live branch and an
`assertNever`-equivalent exhaustiveness guard:

```ts
function composeVersionArrow(from: string, to: string): string {
  return `${from} → v${to}`;
}
```

If the cross-helper defensive contract is genuinely needed, document the
*caller's* preconditions rather than emitting fictional fallback output.

### WR-04: Hand-rolled `getAllTools()`-based soft-dep probe is not exercised against real Pi tool shapes

**File:** `extensions/pi-claude-marketplace/platform/pi-api.ts:51-78` (consumed via `shared/notify.ts:1119`)
**Issue:** `hasLoadedPiSubagents` matches a tool whose `name === "subagent"`.
`hasLoadedPiMcpAdapter` matches `name === "mcp"` OR `sourceInfo.source`
substring-includes `"pi-mcp-adapter"`. The tests in `tests/shared/notify-v2.test.ts`
and `tests/architecture/catalog-uat.test.ts` mock `pi.getAllTools()` with
plain objects (`{ name: "subagent" }`, `{ name: "mcp" }`); they never exercise
the substring fallback for `pi-mcp-adapter`, never exercise the `try/catch`
that degrades to `false` on probe failure, and never exercise the case where
two different tools share the matched name. If Pi's `getAllTools()` shape
drifts (e.g. tool names get prefixed, or `sourceInfo` gains nested objects),
the probe degrades silently to "unloaded" and every `{requires pi-subagents}`
marker fires forever. There is no integration test in this review scope.
**Fix:** Add a unit test in `tests/platform/pi-api.test.ts` (or extend
`tests/shared/notify-v2.test.ts`) that covers (a) the `pi-mcp-adapter`
substring path, (b) the `try/catch` fallback when `getAllTools()` throws, (c)
the boundary case where `tool.name` is `undefined`. None of these branches is
currently covered.

### WR-05: `piWithSubagentsLoaded` referenced via `void` to mask unused-helper warning

**File:** `tests/architecture/catalog-uat.test.ts:1328-1332`
**Issue:** The fixture file defines four MockPi factories but no current
catalog state uses `piWithSubagentsLoaded` (the pi-mcp-adapter-not-loaded
counterpart of `piWithMcpLoaded`). The file works around this with
`void piWithSubagentsLoaded;` and a comment claiming the function "remains
available as a composition primitive for future states". Two problems:
(1) dead code is dead code; if no fixture needs it, deleting it is the
correct response and a future fixture author can re-add it in 3 lines.
(2) The `void` expression silences the lint warning but does not exercise
the function -- it is effectively a `@ts-ignore` for a real signal. If a
future refactor breaks `piWithSubagentsLoaded`'s body, no test catches it.
**Fix:** Delete `piWithSubagentsLoaded` plus the `void` hack. If a later
catalog state needs the inverse-of-mcp-loaded probe, re-add it then.

### WR-06: `default: return assertNever(mp.status)` is unreachable but typechecks for the wrong reason

**File:** `extensions/pi-claude-marketplace/shared/notify.ts:623-625`
**Issue:** The `renderMpHeader` switch has arms for all 7
`MarketplaceStatus` literals AND `case undefined:`, so `mp.status` is
narrowed to `never` at the `default:` arm. `assertNever(mp.status)` is the
correct exhaustiveness gate -- BUT the `return assertNever(...)` shape relies
on `assertNever`'s `never` return type rather than its `throw` behaviour for
the typecheck. If a future edit drops one of the literal cases (e.g. removes
`case "skipped":` while keeping the others), the typechecker correctly
flags it. However, if a future edit *adds* a new `MarketplaceStatus` literal
to `MARKETPLACE_STATUSES` and forgets to add the `case` arm, the
**arch-test** in `tests/architecture/notify-types.test.ts:112-115` will *not*
fail (it only asserts length === 7) and the runtime path falls into
`assertNever`, which throws at notify time. This is acceptable but inverts
the "compile-error gate" claim in the docstring at line 624. The pair of
checks (length === 7 + per-literal forward/backward extension at lines 142-155)
*does* catch the addition at the type level -- but only if the developer
remembers to bump the expected literal union at lines 142-149 alongside the
tuple. The arch test relies on a manually-maintained mirror; the renderer's
exhaustiveness gate is the only true compile-time check.
**Fix:** Either (a) add a comment to `MARKETPLACE_STATUSES` instructing
authors to also update the renderer switch when extending the tuple, or
(b) restructure the test to derive the expected union from the tuple itself
(but then the test no longer detects rename drift). The current shape is
acceptable; this finding records the asymmetry for future maintainers.

### IN-03: `notifyUsageError` V1 3-arg overload tolerates `usageBlock === undefined` at runtime

**File:** `extensions/pi-claude-marketplace/shared/notify.ts:110-131`
**Issue:** The unified implementation signature accepts
`usageBlock?: string`, and the V1 path emits
`` `${message}\n\n${usageBlock ?? ""}` ``. If a JavaScript caller bypasses the
TypeScript overload signature and passes only two args (`message: string`),
the output becomes `"<message>\n\n"` -- a usage error with a blank usage
section. The V1 overload signature requires `usageBlock: string` so this is
type-protected for sanctioned callers, but the runtime fallback silently
emits a malformed body instead of raising. Phase 21 will delete the V1
3-arg overload entirely, so this is a short-lived footgun, but during the
migration window a bridge or orchestrator that lost its `usageBlock` will
emit an empty usage section without anyone noticing.
**Fix:** Either throw on `typeof message === "string" && usageBlock === undefined`
or drop the `?? ""` fallback so TypeScript's strict-null-check catches the
omission. Optional now that V2 callers don't take this path -- defer to
Phase 21 deletion if preferred.
