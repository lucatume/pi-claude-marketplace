# Phase 17.1: V2 Grammar Amendment: Autoupdate Surface - Context

**Gathered:** 2026-05-26
**Status:** Ready for planning

<domain>
## Phase Boundary

Amend the V2 notify grammar (type model + renderer + catalog + ADR) to support the user-locked autoupdate enable/disable surface design from D-18-05 (Phase 18's discuss-phase). Phase 17's v2 catalog had collapsed `marketplace autoupdate enable|disable` to a single `(updated)` status with no idempotent-flip distinction; that erased the V1 user-visible distinction between fresh enable/disable, idempotent no-ops, and failures. Phase 17.1 restores the distinction by adding three new `MarketplaceStatus` values, an optional `reasons?:` field on `MarketplaceNotificationMessage`, three new `renderMpHeader` arms, an updated severity/reload-hint ladder, a rewritten catalog autoupdate section with five per-state fixtures, and an amended ADR Decision section. Phase 17.1 is a pure layered amendment -- no new orchestrator behavior; the call-site migration of `orchestrators/marketplace/autoupdate.ts` is Plan 18-02 (downstream).

**In scope (this phase):**

1. **Type model (Phase 15 amendment):** Add `"autoupdate enabled"`, `"autoupdate disabled"`, `"skipped"` to `MARKETPLACE_STATUSES` (4 → 7); add optional `reasons?: readonly Reason[]` to `MarketplaceNotificationMessage`. Update `tests/architecture/notify-types.test.ts` length assertion (4 → 7), literal-union assertion (`_MarketplaceStatusExpected` add 3), and `_MarketplaceMessageExpected` shape (add `readonly reasons?: readonly Reason[]`).
2. **Renderer (Phase 16 amendment):** Add three new arms to `renderMpHeader` switch (`"autoupdate enabled"`, `"autoupdate disabled"`, `"skipped"`); the `"skipped"` arm renders `${ICON_INSTALLED} ${name} [${scope}] (skipped) {<reasons>}` consuming `mp.reasons`. Update `computeSeverity` to route `"skipped"` (marketplace-level) to warning consistent with plugin-level `"skipped"` per D-16-11. Update `shouldEmitReloadHint` to fire on `"autoupdate enabled"` and `"autoupdate disabled"` (fresh state-change) and NOT on `"skipped"` (no state change). Add five new byte-equality unit tests to `tests/shared/notify-v2.test.ts`: three per-arm tests + one severity ladder test + one reload-hint ladder test.
3. **Catalog (Phase 17 amendment):** Rewrite the `## /claude:plugin marketplace autoupdate <enable|disable> <name>` section of `docs/output-catalog.md` (lines 838-882). Replace the existing `enable-mixed` / `disable-mixed` / `failure-not-found` fixtures with five new per-state fixtures: `enable-fresh`, `disable-fresh`, `enable-idempotent`, `disable-idempotent`, `failure-not-found`. Full prose rewrite (the existing `"v1.0 idempotent-flip variant ... is not expressible in v2"` paragraph is now factually wrong). Update the corresponding fixture map in `tests/architecture/catalog-uat.test.ts` (lines 1244-1278) to match. Update the `MarketplaceStatus; // 4 literal strings` pointer in `docs/messaging-style-guide.md` to `// 7 literal strings`.
4. **ADR amendment:** Update `docs/adr/v2-001-structured-notify.md` Decision section in place (lines 35, 42, 88) from 4-entry to 7-entry `MARKETPLACE_STATUSES`. Add a new `## Amendment: Phase 17.1 (2026-MM-DD)` section at the bottom describing what changed and why (preserves historical context while marking the current state). Consequences / Migration / Alternatives sections stay untouched.

**Out of scope (not Phase 17.1):**

- **Migrating `orchestrators/marketplace/autoupdate.ts`** -- Plan 18-02 (downstream consumer of this phase's amendments).
- **Modifying any other marketplace/* or plugin/* orchestrator** -- outside the autoupdate surface.
- **`shared/grammar/reasons.ts` additions** -- `"already enabled"` and `"already disabled"` are ALREADY present at lines 65-66 (verified during scout); no source file change required for the reasons set. The binding frontmatter in `docs/messaging-style-guide.md` may need a refresh if any drift test still asserts on the 23-entry count.
- **Refreshing the ADR Consequences / Migration / Alternatives sections** -- only the Decision section is amended in place; the rest of the ADR's accepted narrative stays intact.
- **Phase 18's plan 18-02 unblock signal** -- Phase 17.1's `VERIFICATION.md status: passed` is the gate Plan 18-02 reads; 17.1 produces the verification artifact; 18-02 consumes it. No mechanism added in 17.1.
- **`marketplace list` `<autoupdate>` / `<no autoupdate>` markers** -- the list-surface header arm (`mp.status === undefined` + `mp.details.autoupdate === true`) is unchanged. State-change arms (`autoupdate enabled` / `autoupdate disabled`) do NOT carry the angle-bracket marker; the marker is reserved for the list surface.

<decisions>
## Implementation Decisions

### Plan granularity & wave order

- **D-17.1-01:** Phase 17.1 ships **4 plans across 3 waves**. Wave 1 = `17.1-01` (type model + arch tests). Wave 2 = `17.1-02` (renderer + notify-v2.test.ts byte-equality unit tests). Wave 3 = `17.1-03` (catalog rewrite + UAT fixtures + messaging-style-guide pointer update) and `17.1-04` (ADR amendment). The two Wave 3 plans are independent of each other and can land in parallel. Per-layer granularity matches Phase 16's per-arm precedent and gives each layer an atomic commit boundary (one source-of-truth surface per plan).
- **D-17.1-02:** Wave order is data-dependency-driven, not narrative. `17.1-03` (catalog + UAT) strictly depends on `17.1-02` (renderer) because the UAT test drives `notify()` through the new renderer arms -- without those arms the byte-equality fixtures throw (`assertNever`) or fail to compile (closed-set type widening). The "parallel Wave 2" alternative was rejected to avoid a mid-wave byte-equality break window.

### Catalog fixture migration

- **D-17.1-03:** The `tests/architecture/catalog-uat.test.ts` autoupdate fixture map (lines 1244-1278) is REPLACED with exactly the five D-18-05 states. Each fixture is single-marketplace, demonstrating the precise byte form for that state. Multi-marketplace cascade is already covered structurally by other catalog sections (e.g., `marketplace remove` partial); no `*-mixed` fixture in the autoupdate section. The five fixtures:

  | Fixture key | Payload | Expected bytes | Severity | Reload-hint |
  |---|---|---|---|---|
  | `enable-fresh` | `{ status: "autoupdate enabled", name: "foo", scope: "user", plugins: [] }` | `● foo [user] (autoupdate enabled)\n\n/reload to pick up changes` | undefined (info) | yes |
  | `disable-fresh` | `{ status: "autoupdate disabled", name: "foo", scope: "user", plugins: [] }` | `● foo [user] (autoupdate disabled)\n\n/reload to pick up changes` | undefined (info) | yes |
  | `enable-idempotent` | `{ status: "skipped", reasons: ["already enabled"], name: "foo", scope: "user", plugins: [] }` | `● foo [user] (skipped) {already enabled}` | `"warning"` | no |
  | `disable-idempotent` | `{ status: "skipped", reasons: ["already disabled"], name: "foo", scope: "user", plugins: [] }` | `● foo [user] (skipped) {already disabled}` | `"warning"` | no |
  | `failure-not-found` | `{ status: "failed", name: "missing-mp", scope: "user", plugins: [] }` | `⊘ missing-mp [user] (failed)` | `"error"` | no |

- **D-17.1-04:** `docs/output-catalog.md` `## /claude:plugin marketplace autoupdate <enable|disable> <name>` section gets a **full prose rewrite**. The existing prose contains statements that are factually wrong after the amendment ("v1.0 idempotent-flip variant ... not expressible in v2", "Same structural shape as enable-mixed", "the orchestrator chooses (updated) for both directions"). Replace with new prose describing the five-state mapping, the severity ladder (fresh → info, skipped → warning, failed → error), and the reload-hint ladder (fresh triggers, skipped/failed suppress). Keep the section's intro one-liner ("marketplace-only flag flip ... no plugin children") and the cross-reference to the list-surface marker convention.

### Reusable Assets

- **`renderMpHeader` switch (`shared/notify.ts:529-567`)** -- File-private renderer with `assertNever(mp.status)` exhaustiveness at line 565. Adding three new arms is a mechanical extension -- the existing pattern (4 arms returning a template-literal string) is the template; `"skipped"` is the only new arm that composes `mp.reasons` (other 6 arms don't carry reasons). Plan 17.1-02 reads this entire switch before adding arms.
- **`composeReasons` helper (`shared/notify.ts:684-705`)** -- Already file-private and consumed by `renderPluginRow`. The `"skipped"` mp-header arm can reuse this helper to compose `{<reason>, <reason>}` from `mp.reasons` (with `false, false` for the soft-dep markers -- mp-level skipped never adds soft-dep reasons; those are plugin-level only).
- **`computeSeverity` / `shouldEmitReloadHint` (`shared/notify.ts:903-950`)** -- Pure functions reading the message structurally. Plan 17.1-02's amendments are localized to these two functions: adding `mp.status === "skipped"` to the second-match pass of `computeSeverity`, and adding `"autoupdate enabled"` / `"autoupdate disabled"` to the first-match pass of `shouldEmitReloadHint`.
- **`tests/shared/notify-v2.test.ts` per-variant pattern** -- Phase 16's 32 tests are organized per-status (each `describe("status X", …)` block tests the byte form, soft-dep marker behavior, severity contribution, reload-hint contribution). Plan 17.1-02's 5 new tests follow this pattern.

### Established Patterns

- **Renderer-as-spec discipline (D-16-04)** -- Phase 17.1's renderer changes ARE the new grammar. The catalog block in `docs/output-catalog.md` is the human-readable spec; the renderer arm is the machine-checked spec; the byte-equality assertion in catalog-uat.test.ts ties them together. All three must move atomically (across the 17.1-02 → 17.1-03 wave boundary).
- **`as const` tuple + indexed-access derived type pattern (D-15-11)** -- Adding to `MARKETPLACE_STATUSES` automatically widens `MarketplaceStatus`; the only loose end is the architecture test's `_MarketplaceStatusExpected` literal union and `_Assert_MarketplaceStatusesLen` count assertion, both of which Plan 17.1-01 updates explicitly.
- **Optional-independent fields discipline (D-15-06)** -- `reasons?:` is the THIRD independent optional on `MarketplaceNotificationMessage` (alongside `status?:` and `details?:`). Adding it follows the same pattern; the type itself does not constrain co-occurrence (renderer enforces it: only the `"skipped"` arm consumes `mp.reasons`; other arms ignore it).
- **ADR Decision-section edit-in-place vs. Amendment section (D-17.1-07)** -- Standard pattern for closed-set extensions: update the count + literals in Decision (the "current state" surface) AND append an `## Amendment: Phase N.M` section (the "what changed and why" surface). Both surfaces live in the same file.

## Specific Ideas

- **The five catalog fixture names are normative** -- `enable-fresh`, `disable-fresh`, `enable-idempotent`, `disable-idempotent`, `failure-not-found`. These names appear in both `docs/output-catalog.md` (`<!-- catalog-state: ... -->` discriminators) and `tests/architecture/catalog-uat.test.ts` (fixture map keys). Plans 17.1-03 must keep them in lockstep.
- **The 5-state byte forms in D-17.1-03 are normative** -- exact bytes including the `\n\n/reload to pick up changes` trailer (or its absence) per the severity/reload-hint ladders. Plan 17.1-02's renderer arms + ladders must produce these bytes when fed the payloads in the table; Plan 17.1-03's catalog blocks must contain these bytes verbatim.
- **The ADR Amendment section follows a fixed structure** -- heading `## Amendment: Phase 17.1 (YYYY-MM-DD)`, then 3 short paragraphs: (a) what changed (3 statuses + optional reasons field); (b) why (V1 distinction lost in Phase 17, user-locked in D-18-05); (c) ladder updates (severity / reload-hint). The planner can phrase these paragraphs but must cover all three.
- **The `"skipped"` mp-status arm reuses `composeReasons`** -- to maintain consistency with how plugin-level `"skipped"` renders reasons. The mp arm passes `false, false` for the two soft-dep marker booleans (mp-level skipped never emits soft-dep markers; those are plugin-row-only per D-16-15).
