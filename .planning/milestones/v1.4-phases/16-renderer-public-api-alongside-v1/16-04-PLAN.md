<objective>
Add a file-private `renderPluginRow(p: PluginNotificationMessage, probe: SoftDepStatus): string` helper to `extensions/pi-claude-marketplace/shared/notify.ts`. This is the second of two file-private rendering helpers Phase 16 ships (D-16-09 second helper; first was `renderMpHeader` in plan 03). The helper switches on the full `PluginNotificationMessage` discriminated union over its 10 variants, narrows on `p.status`, and returns a single-line plugin-row string. Soft-dep markers (`{requires pi-subagents}` / `{requires pi-mcp}`) are computed at render time from the per-row `dependencies?: readonly Dependency[]` and the threaded `SoftDepProbe`. Plan 05 will compose this helper into the public `notify()` and supply the indent-prefix; this plan returns the bare row content (no leading two-space indent -- caller's job).

Per-variant Phase 15 type model (read directly from `extensions/pi-claude-marketplace/shared/notify.ts:269-403` -- this is the AUTHORITATIVE field-availability map and every per-arm rule below MUST match it):

| Variant | reasons | dependencies | version? | scope? | cause? | rollbackPartial? |
|---|---|---|---|---|---|---|
| installed | -- | REQUIRED | yes | yes | -- | -- |
| updated | -- | REQUIRED | NO (carries `from`/`to`) | yes | -- | -- |
| reinstalled | -- | REQUIRED | yes | yes | -- | -- |
| uninstalled | -- | -- | yes | yes | -- | -- |
| available | -- | -- | yes | NO | -- | -- |
| unavailable | REQUIRED | -- | yes | NO | -- | -- |
| upgradable | REQUIRED | -- | yes | yes | -- | -- |
| failed | REQUIRED | -- | yes | yes | yes | yes |
| skipped | REQUIRED | -- | yes | yes | -- | -- |
| manual recovery | REQUIRED | -- | yes | yes | yes | -- |

Key implications enforced below:
  - `reasons: readonly Reason[]` is REQUIRED on 5 variants (unavailable / upgradable / failed / skipped / manual recovery) and ABSENT on the other 5. Per-arm rules MUST pass `undefined` (not `p.reasons`) for the variants that lack the field.
  - `dependencies: readonly Dependency[]` is REQUIRED on installed / updated / reinstalled and ABSENT on the other 7.
  - `scope?: Scope` is OPTIONAL on 8 variants (all except available / unavailable) -- the `[${p.scope}]` bracket MUST be emitted conditionally on `p.scope !== undefined`. In practice the bracket appears only in the orphan-fold case per CONTEXT D-16-17 + SNM-11; the marketplace header's scope still wins for the header itself, and the plugin row's scope appears inline only when it differs from (or is independently set on) the per-row payload. The `available` / `unavailable` variants have NO `scope` field at all -- their per-arm rules unconditionally omit the bracket (MSG-PL-6 / SNM-11 carve-out).
  - `version?: string` is OPTIONAL on 9 variants (all except `updated`, which carries REQUIRED `from`/`to`).
  - `cause?: Error` is OPTIONAL on `failed` / `manual recovery` only; `rollbackPartial?` on `failed` only. Neither is rendered by `renderPluginRow` -- plan 05's `notify()` composes the indented cause-chain and rollback-partial trailer lines after the row.

Purpose: SNM-17 requires the internal switch over plugin `status` to be the sole site that knows the v2 plugin-row grammar. SNM-16 requires the soft-dep probe markers to be computed at render time (no caller-supplied probe state). D-16-09 extracts this into a file-private helper with its own `assertNever(p)` exhaustiveness gate so adding any new `PluginNotificationMessage` variant becomes a compile error at this switch.
