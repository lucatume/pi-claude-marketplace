# Phase 79.1: Mutable mirror clones for unpinned git plugin sources - Context

**Gathered:** 2026-07-13
**Status:** Ready for planning

<domain>
## Phase Boundary

An unpinned git-source plugin (url / git-subdir / github with **no manifest `sha`** — ref-only sources are moving pointers and count as unpinned) is backed by exactly ONE marketplace-style mutable mirror clone per canonical URL at `plugin-clones/<urlhash12>/` (no sha suffix), refreshed in place like a marketplace clone. Fetched-state derives from mirror-dir presence; multi-clone ambiguity is impossible by construction. Pinned sources keep the per-sha immutable content-addressed cache (`plugin-clones/<urlhash12>-<sha12>/`) byte-identically to today.

Covers MIRR-01..06. Downstream: Phase 80 classifies `(remote)` from mirror/exact-key absence; Phase 81's `fetch` verb on an unpinned source = refresh the mirror.

</domain>

<decisions>
## Implementation Decisions

### Design principle: marketplace parity, reuse over reinvention (LOCKED)
- **D-79.1-01:** The mirror lifecycle reuses the marketplace clone machinery — same semantics AND same code wherever it maps. Operator's directive: "can we apply the same semantics and techniques and choices and much of the same code... that way we wouldn't need to reinvent ideas." Mapping:
  - First materialization = marketplace-add flow: staging clone → atomic rename (the `addGitClonedInGuard` pattern already distilled into `materializePluginClone`), target `plugin-clones/<urlhash12>/`.
  - Refresh = `refreshGitHubClone(cloneDir, ref, gitOps, onFetchSucceeded, auth)` — the SAME function marketplace update uses (in-place fetch + force-update ref + checkout, `cloneAdvanced` fail-state tracking, host-agnostic). ref = `source.ref` when present, else default HEAD.
  - Auth = shared provider registry (`buildAuthForHost`, once-per-host memo) — already shared, no change.
  - Containment = SC-7 chokepoint (`pluginCloneDir`) — already in place.
  - Cleanup = existing `garbageCollectPluginClones` unchanged: its first-path-segment key derivation protects a mirror (`<urlhash12>/…`) exactly like a per-sha clone.
  Only true deltas from marketplaces (all domain-forced): keyed by URL-hash not user-chosen name (plugin sources have no name; one URL dedups across marketplaces), derive-not-persist lifecycle (GC sweep instead of explicit add/remove state records), and the consumer is the resolver/bridges reading plugin trees instead of `marketplace.json`.

### Refresh policy
- **D-79.1-02:** Mirrors refresh exactly where marketplace clones refresh — on explicit mutating verbs only: install / update / reinstall now, Phase 81 `fetch` later. Read surfaces (`list`, bare `info`, completion) read the warm mirror fs-only and NEVER refresh — network policy unchanged (NFR-5). This preserves today's unpinned semantics (install always resolves remote state) while keeping reads network-free.

### Concurrency / atomicity
- **D-79.1-03:** Marketplace parity, no new machinery: same in-place refresh sequence and the same accepted multi-process exposure marketplace clones ship with today. Fail-clean single-process behavior via the existing `cloneAdvanced`-style tracking. First materialization keeps staging + atomic rename (NFR-1 applies to the create path; the refresh path inherits the marketplace-update precedent).

### Migration / coexistence with the old per-sha design
- **D-79.1-04:** Lazy re-anchor via lifecycle ops (no load-time backfill, no persisted migration state). Installed unpinned plugins keep their recorded per-sha clone references (`<urlhash12>-<sha12>/…`) — GC keeps protecting them — until the next update/reinstall of that plugin naturally re-materializes through the mirror and rewrites the record. Orphaned per-sha clones of never-reanchored plugins sweep via GC after uninstall, as today.

### Claude's Discretion
- Whether to extract a shared clone-lifecycle module both marketplace and mirror paths import, vs. calling the marketplace helpers across — planner decides by import-graph/architecture-gate fit (the no-orchestrator-network gate must stay green; mirror refresh lives in a git-legal seam like `clone-cache.ts`, never in list/info paths).
- Mirror-vs-per-sha key shape discrimination (`<12hex>` vs `<12hex>-<12hex>`) — mechanical; GC needs no change but any key-shape assertions must accept both.
- Exact record shape for a mirror-anchored install (`resolvedSource` points into the mirror; `resolvedSha` = sha read from the mirror at install time, PURL-09/PI-7 preserved).

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `refreshGitHubClone`: host-agnostic in-place refresh — the mirror refresh IS this function.
- `materializePluginClone`: staging→atomic-rename create; generalize the key/target for the mirror case.
- `garbageCollectPluginClones`: works unchanged — mirror keys derive from `resolvedSource` first segment like per-sha keys.
- Provider auth registry (`buildAuthForHost`): already shared between marketplace and plugin clone paths.

### Established Patterns
- No-orchestrator-network architecture gate: list/info/completion modules must stay free of the git surface; mirror refresh belongs in the git-legal clone-cache seam.
- Derive-not-persist: mirror-dir existence IS the fetched-state; no registry, no refcount, no migration stamp.
- SC-7: every cache path routes through `pluginCloneDir` (assertSafeName + assertPathInside) before any read/delete.

### Integration Points
- `install.ts` `cloneCacheSeam` — the injection point where unpinned sources route to mirror materialize/refresh instead of per-sha materialize.
- `resolvePluginPin` — pin resolution (sha > ref > HEAD) stays; for unpinned sources the resolved sha is read from the refreshed mirror rather than driving a per-sha cache key.
- Phase 80's `git-source-probe.ts` classification will consume mirror presence for the unpinned arm (RSTA-06 as rewritten).

</code_context>
