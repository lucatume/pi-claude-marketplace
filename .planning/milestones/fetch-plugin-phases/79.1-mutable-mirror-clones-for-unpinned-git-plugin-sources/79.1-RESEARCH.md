# Phase 79.1: Mutable mirror clones for unpinned git plugin sources - Research

**Researched:** 2026-07-13
**Domain:** Internal codebase — plugin clone-cache lifecycle, git-source install/update/reinstall call graph, marketplace-clone parity, GC derive-not-persist
**Confidence:** HIGH (all findings are direct source reads of the target codebase; no external library research required)

## Summary

This phase inserts a second clone shape into the existing plugin clone cache: unpinned git-source plugins (url / git-subdir / github with no manifest `sha`, plus ref-only sources) get exactly ONE mutable mirror clone per canonical URL at `plugin-clones/<urlhash12>/` (bare left-half key, no `-<sha12>` suffix), refreshed in place with the SAME `refreshGitHubClone` machinery marketplace update already uses. Pinned sources keep the existing per-sha immutable content-addressed cache `plugin-clones/<urlhash12>-<sha12>/` byte-identically. The core machinery to reuse already exists and is proven: `refreshGitHubClone` (in-place fetch + force-update-ref + checkout, D-14 sequence), `materializePluginClone` (staging → atomic-rename create), the shared `buildAuthForHost` registry, the SC-7 `pluginCloneDir` chokepoint, and `garbageCollectPluginClones` (first-path-segment key derivation, which already protects a bare `<urlhash12>` mirror dir the same way it protects a per-sha dir). [VERIFIED: source reads of clone-cache.ts, shared.ts, clone-gc.ts, locations.ts]

The single hardest constraint is the `no-orchestrator-network` architecture gate (`tests/architecture/no-orchestrator-network.test.ts`): it forbids `install.ts`, `reinstall.ts`, `list.ts`, and both `info.ts` files from importing `gitOps` / `platform/git` / `DEFAULT_GIT_OPS` / referencing `refreshGitHubClone`. `refreshGitHubClone` currently lives in `orchestrators/marketplace/shared.ts`. The mirror refresh MUST live in a git-legal seam — `clone-cache.ts` is git-legal (imports `DEFAULT_GIT_OPS` from `marketplace/shared.ts`, is NOT in the gate's forbidden list) and is the sanctioned home. `install.ts` reaches it by name through the injected `cloneCacheSeam` (`resolvePluginPin` / `materializePluginClone`), exactly as today. `update.ts` is the one gitOps-exempt plugin orchestrator (Pattern S-9). [VERIFIED: no-orchestrator-network.test.ts FORBIDDEN_TARGETS + FORBIDDEN_PATTERNS]

**Primary recommendation:** Add a mirror arm to `clone-cache.ts` — a `pluginMirrorKey(canonicalUrl)` (bare 12-hex left half) in `domain/clone-key.ts`, a `materializePluginMirror` / `refreshPluginMirror` pair in `clone-cache.ts` that reuses `refreshGitHubClone` + the `materializePluginClone` staging→rename create path, and read the checked-out sha from the mirror working tree with `gitOps.resolveRef({ dir, ref: "HEAD" })` after refresh (git-legal, only inside clone-cache.ts). Fork the unpinned branch inside the install/update/reinstall clone probes (`makeInstallCloneProbe` / `makeUpdateCloneProbe` / `makeReinstallCloneProbe`), NOT in the orchestrator bodies. Extend `makePresenceProbe` (git-source-probe.ts) so an unpinned source derives presence from the mirror dir instead of returning `not-cached`.

## User Constraints (from CONTEXT.md)

### Locked Decisions

- **D-79.1-01 (marketplace parity, reuse over reinvention):** The mirror lifecycle reuses the marketplace clone machinery — same semantics AND same code wherever it maps.
  - First materialization = marketplace-add flow: staging clone → atomic rename (the `addGitClonedInGuard` pattern already distilled into `materializePluginClone`), target `plugin-clones/<urlhash12>/`.
  - Refresh = `refreshGitHubClone(cloneDir, ref, gitOps, onFetchSucceeded, auth)` — the SAME function marketplace update uses (in-place fetch + force-update ref + checkout, `cloneAdvanced` fail-state tracking, host-agnostic). ref = `source.ref` when present, else default HEAD.
  - Auth = shared provider registry (`buildAuthForHost`, once-per-host memo) — already shared, no change.
  - Containment = SC-7 chokepoint (`pluginCloneDir`) — already in place.
  - Cleanup = existing `garbageCollectPluginClones` unchanged: its first-path-segment key derivation protects a mirror (`<urlhash12>/…`) exactly like a per-sha clone.
  - True deltas from marketplaces (all domain-forced): keyed by URL-hash not user-chosen name; derive-not-persist lifecycle (GC sweep, no explicit add/remove state records); consumer is the resolver/bridges reading plugin trees, not `marketplace.json`.
- **D-79.1-02 (refresh policy):** Mirrors refresh on explicit mutating verbs only — install / update / reinstall now, Phase 81 `fetch` later. Read surfaces (`list`, bare `info`, completion) read the warm mirror fs-only and NEVER refresh (NFR-5 unchanged).
- **D-79.1-03 (concurrency / atomicity):** Marketplace parity, no new machinery — same in-place refresh sequence and the same accepted multi-process exposure marketplace clones ship with today. Fail-clean single-process via existing `cloneAdvanced`-style tracking. First materialization keeps staging + atomic rename (NFR-1 on the create path; refresh inherits the marketplace-update precedent).
- **D-79.1-04 (migration / coexistence):** Lazy re-anchor via lifecycle ops — no load-time backfill, no persisted migration state. Installed unpinned plugins keep their recorded per-sha clone references (`<urlhash12>-<sha12>/…`); GC keeps protecting them until the next update/reinstall re-materializes through the mirror and rewrites the record. Orphaned per-sha clones sweep via GC after uninstall.

### Pattern 1: Marketplace-parity in-place refresh (the parity anchor)

**What:** The mirror refresh is `refreshGitHubClone` verbatim — the D-14 three-step follow-upstream-blindly sequence, host-agnostic, with `cloneAdvanced` fail-state tracking through the `onFetchSucceeded` callback.
**When to use:** Refreshing an already-present mirror on a mutating verb.
**Example:**
```typescript
// Source: orchestrators/marketplace/shared.ts (refreshGitHubClone) + update.ts refreshRecord
// storedRef = source.ref (branch/tag) or undefined (default HEAD).
let cloneAdvanced = false;
await refreshGitHubClone(
  mirrorDir,
  source.ref,        // undefined => default-branch HEAD tracking (D-14 first form)
  gitOps,
  () => { cloneAdvanced = true; },   // MU-5: flip only AFTER fetch returns
  authBundle,
);
// Read the newly checked-out sha for the PURL-09 recorded version (git-legal here):
const pin = await gitOps.resolveRef({ dir: mirrorDir, ref: "HEAD" });
```

### Pattern 2: Staging → atomic rename for first materialization (create path)

**What:** First mirror materialization mirrors `materializePluginClone`'s create path: clone into `sourcesStagingDir(uuid)`, then `rename()` onto `pluginCloneDir(mirrorKey)`. An EEXIST/ENOTEMPTY rename means a concurrent op won the race → clean staging, treat as warm-cache win.
**When to use:** Mirror dir absent on a mutating verb.
**Example:**
```typescript
// Source: orchestrators/plugin/clone-cache.ts::materializePluginClone (lines 74-113)
const stagingDir = await locations.sourcesStagingDir(randomUUID());
try {
  await gitOps.clone({ dir: stagingDir, url: cloneUrl,
    ...(ref !== undefined && { ref, singleBranch: true }),
    ...(auth !== undefined && { auth }) });
  // For a mirror, do NOT checkout a fixed pin — the working tree tracks HEAD/ref.
} catch (err) {
  const leak = await cleanupStaging(stagingDir, "plugin mirror staging");
  throw appendLeakToError(err, leak);
}
await mkdir(path.dirname(mirrorRoot), { recursive: true });
await rename(stagingDir, mirrorRoot);   // EEXIST/ENOTEMPTY => concurrent win
```

### Pattern 3: Fork-in-the-probe, never in the orchestrator body

**What:** The unpinned→mirror branch lives inside `makeInstallCloneProbe` / `makeUpdateCloneProbe` / `makeReinstallCloneProbe` — the `resolveGitPluginRoot` callback the resolver invokes. The orchestrator bodies (`runInstallLedger`, etc.) never name the git surface.
**When to use:** Always — this is what keeps install.ts/reinstall.ts gate-green.
**Example:**
```typescript
// Source: orchestrators/plugin/install.ts::makeInstallCloneProbe (lines 488-546)
const probe = async (gitSource) => {
  if (gitSource.sha === undefined) {          // NEW unpinned arm
    const { cloneUrl } = ...;                  // canonicalCloneUrl(gitSource)
    const { pluginRoot, pin } = await seam.materializeOrRefreshMirror({ locations, cloneUrl, ref, auth });
    captured = pin;
    // git-subdir: resolveGitSubdirRoot(mirrorRoot, gitSource.path) as today
    return { kind: "materialized", pluginRoot, resolvedSha: pin };
  }
  // existing per-sha arm: resolvePluginPin + materializePluginClone
};
```

### Anti-Patterns to Avoid

- **Importing `refreshGitHubClone` / `gitOps` into install.ts or reinstall.ts:** Instant `no-orchestrator-network.test.ts` failure. The gate greps stripped source for `\brefreshGitHubClone\b`, `\bgitOps\b`, `\bDEFAULT_GIT_OPS\b`, and `from platform/git`. Keep all git surface in clone-cache.ts, reached via the `cloneCacheSeam` bundle by name.
- **Deriving the mirror sha offline in a read surface via network:** `makePresenceProbe` is fs-only (NFR-5). Read the mirror sha from the on-disk working tree, never via `resolveRemoteRef`. (See Pitfall 3 for the fs-only read.)
- **Checking out a fixed pin into the mirror:** The mirror tracks a moving ref/HEAD; the per-sha cache checks out an immutable pin. Do not conflate — the mirror's whole point is in-place mutability.
- **Adding a persisted migration stamp / refcount:** Violates derive-not-persist (D-79.1-04). Fetched-state IS the mirror dir's existence; live-key protection IS the first-path-segment of `resolvedSource`.

## Common Pitfalls

### Pitfall 1: Breaking the no-orchestrator-network gate
**What goes wrong:** Adding a mirror refresh helper that imports `refreshGitHubClone` or `gitOps` into `install.ts` / `reinstall.ts` / `list.ts` / `info.ts`.
**Why it happens:** The natural instinct is to call the refresh where the fork decision is made (the orchestrator body).
**How to avoid:** Put the mirror materialize+refresh helper in `clone-cache.ts` (git-legal). Expose it through the existing `InstallCloneCacheSeam` / `UpdateCloneCacheSeam` / `ReinstallCloneCacheSeam` bundle so orchestrators reach it by name. The fork (`source.sha === undefined`) lives inside the `makeXxxCloneProbe` callback, which already legally receives the seam.
**Warning signs:** `tests/architecture/no-orchestrator-network.test.ts` fails with an offender line naming install.ts/reinstall.ts and pattern `refreshGitHubClone` or `gitOps`.

### Pitfall 2: `list` / bare `info` / completion accidentally refreshing (NFR-5)
**What goes wrong:** The mirror-presence probe triggers a network refresh on a read surface.
**Why it happens:** Sharing a single "materialize-or-refresh" helper between mutating verbs and read surfaces.
**How to avoid:** Keep two distinct entrypoints. `makePresenceProbe` (git-source-probe.ts) does fs-only `pathExists(pluginCloneDir(mirrorKey))` and NEVER calls the git surface (the module has no git import — verify it stays that way). Refresh is a separate clone-cache entrypoint that only the mutating clone probes call (D-79.1-02).
**Warning signs:** git-source-probe.ts gains a `gitOps` / `DEFAULT_GIT_OPS` import; list/info/completion tests that assert "no plugin-clones dir created, no network" (list.test.ts:2237, info.test.ts:2321) start creating dirs or hanging.

### Pitfall 3: Reading the mirror sha fs-only in the read surface vs. git-legal in the mutating path
**What goes wrong:** `makePresenceProbe` needs `resolvedSha` in its `materialized` result (the `GitPluginRootResult.materialized` arm requires it), but it must NOT spawn git.
**Why it happens:** The mutating path reads the checked-out sha with `gitOps.resolveRef({ dir, ref: "HEAD" })` — but that's git surface, illegal in git-source-probe.ts (which list/completion import).
**How to avoid:** In the fs-only read surface, read the sha directly from the on-disk git dir without the git surface — e.g. read `<mirrorDir>/.git/HEAD` (and follow the ref file to `<mirrorDir>/.git/refs/heads/<branch>`, or use the packed-refs) with plain `fs`. Alternatively, if Phase 80's classifier only needs presence (not the exact sha) for the `(remote)` vs materialized distinction, the read surface can return the mirror pluginRoot with a sha read fs-only. Confirm the resolver's `materialized` arm actually needs the sha at read time, or whether presence alone suffices for RSTA-06 — this is the seam between 79.1 and 80. **[ASSUMED]** that a plain-fs `.git/HEAD` read is the fs-only mechanism; verify isomorphic-git's on-disk layout for the mirror (loose vs packed refs) before locking the read.
**Warning signs:** git-source-probe.ts imports git; or the fs-only read misses a packed-ref-only mirror and returns a stale/empty sha.

### Pitfall 4: EXDEV on the staging→mirror rename
**What goes wrong:** `rename(stagingDir, mirrorRoot)` throws EXDEV if staging and target are on different filesystems.
**Why it happens:** Cross-FS rename is not atomic and fails hard.
**How to avoid:** No new risk — `sources-staging/` and `plugin-clones/` are already siblings under `extensionRoot` (same FS by construction; locations.ts:166-170 documents this for the per-sha path). The mirror target is under the same `pluginClonesDir`, so it inherits the same-FS guarantee. Do not stage anywhere else.
**Warning signs:** EXDEV in materialize; a staging dir placed under a temp/other root.

### Pitfall 5: git-subdir resolution against a mirror root
**What goes wrong:** For a git-subdir source, the pluginRoot is `mirrorRoot + subPath`, and containment must be re-asserted against the mirror root (not the marketplaceRoot).
**Why it happens:** The subdir arm is easy to forget when adding the mirror branch.
**How to avoid:** Reuse `resolveGitSubdirRoot(mirrorRoot, gitSource.path)` exactly as the per-sha probes do (install.ts:530, update.ts:795, reinstall.ts:1353). It already anchors `assertPathInside` to the clone root and returns the `escapes` / `missing-subdir` arms.
**Warning signs:** git-subdir install/update tests fail with containment or missing-subdir on the mirror path.

### Pitfall 6: GC deleting a live mirror because the record's first path segment isn't the bare key
**What goes wrong:** If a mirror-anchored record's `resolvedSource` is written as anything other than `<pluginClonesDir>/<urlhash12>/...`, `deriveLiveCloneKeys` derives the wrong segment and GC sweeps the live mirror.
**Why it happens:** For git-subdir the `resolvedSource` is `<mirrorRoot>/<subdir>` — the first segment relative to `pluginClonesDir` is still `<urlhash12>` (the mirror root), which is correct. But an off-by-one in path construction breaks it.
**How to avoid:** Ensure `resolvedSource` is always `pluginCloneDir(mirrorKey)` (+ subdir for git-subdir). `deriveLiveCloneKeys` takes `path.relative(pluginClonesDir, resolvedSource).split(path.sep)[0]` (clone-gc.ts:44) — verified to yield the bare `<urlhash12>` segment for both the mirror root and the mirror+subdir case. Add a GC test for the mirror shape.
**Warning signs:** clone-gc.test.ts-style test where a mirror-anchored record fails to protect its dir.

1. **RESOLVED — adopted in 79.1-04-PLAN.md Task 1 (warm-mirror-only re-anchor; cold mirror keeps the per-sha record): Reinstall's re-anchor path is network-gated.**
   - What we know: `reinstall.ts` uses `makeReinstallCloneProbe` which pins from the RECORDED sha (`recordedSha`), never calls `resolvePluginPin`/`resolveRemoteRef`, and the no-orchestrator-network gate forbids reinstall.ts from referencing `refreshGitHubClone`/`gitOps` (verified: no-orchestrator-network.test.ts includes reinstall.ts). Does reinstall re-anchor by refreshing an EXISTING mirror only (network-free if warm), or does it skip re-anchor when the mirror is cold and keep the per-sha record?
   - Recommendation: For reinstall, re-anchor to the mirror ONLY when it is already warm (fs-only, PRL-07-compatible); when cold, keep the recorded per-sha reference and let a subsequent `update` (network-legal) do the first materialization. Confirm with the planner whether reinstall re-anchoring is in MIRR-06 scope or deferred to update/install only.

2. **RESOLVED — adopted in 79.1-01-PLAN.md (always refresh-on-warm, marketplace-update parity): Does `materializeOrRefreshPluginMirror` refresh-on-warm, or only materialize-on-cold?**
   - What we know: D-79.1-02 says mutating verbs refresh. `materializePluginClone` today short-circuits on a warm per-sha cache (immutable). A mirror is mutable, so "warm" does NOT mean "skip refresh" for install/update.
