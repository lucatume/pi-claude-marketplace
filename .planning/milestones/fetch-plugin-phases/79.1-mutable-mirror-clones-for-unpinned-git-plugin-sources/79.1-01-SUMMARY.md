# Phase 79.1 Plan 01: Mirror key + materialize/refresh seam Summary

Added the bare URL-keyed mirror key (`pluginMirrorKey`) and the git-legal
mirror materialize+refresh seam (`materializeOrRefreshPluginMirror`) so an
unpinned git plugin source is backed by exactly ONE mutable mirror clone per
canonical URL, refreshed in place at marketplace-clone parity.

## What was built

- **`pluginMirrorKey(canonicalUrl)`** (`domain/clone-key.ts`): returns the bare
  12-hex `sha256(canonicalUrl)` left half Б─■ provably `pluginCloneKey`'s left
  half, with no `-<sha12>` suffix. Reuses the existing `KEY_TRUNC` constant and
  `createHash` import (no re-declaration). Pure Б─■ no network, gitOps, or fs.

- **`materializeOrRefreshPluginMirror(args)`** (`orchestrators/plugin/clone-cache.ts`):
  routes the mirror path through `pluginCloneDir(pluginMirrorKey(cloneUrl))`
  (SC-7 chokepoint). On a cold key it clones into staging (ref-hint
  singleBranch when a ref is given, auth threaded) and atomic-renames to
  `plugin-clones/<urlhash12>/` Б─■ with NO fixed-pin checkout, because the mirror
  tracks a moving HEAD/ref. A clone throw cleans staging and append-leak-rethrows
  (MA-9); an EEXIST/ENOTEMPTY rename is treated as a concurrent-create win
  (clean staging, fall through to refresh). After the dir exists (fresh or warm)
  it calls `refreshGitHubClone(mirrorRoot, ref, gitOps, undefined, auth)` in
  place and reads `resolveRef({ dir, ref: "HEAD" })`, returning
  `{ pluginRoot, resolvedSha }`. All git surface stays confined to
  `clone-cache.ts`.

## How it works

Cold path mirrors `materializePluginClone`'s stagingБ├▓atomic-rename create
(minus the fixed-pin checkout). Warm path is the marketplace update refresh Б─■
the SAME `refreshGitHubClone` D-14 sequence (fetch + force-update ref +
checkout). A just-materialized clone is also refreshed (refresh-on-warm parity,
D-79.1-02). The returned HEAD sha preserves the PURL-09/PI-7 resolved-sha
contract so install/update record the pin.
