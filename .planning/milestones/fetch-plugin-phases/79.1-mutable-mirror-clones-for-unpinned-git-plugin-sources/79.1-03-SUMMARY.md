**Unpinned git plugin install and update now fork inside their clone probes to the `materializeOrRefreshPluginMirror` seam, recording the bare `plugin-clones/<urlhash12>/` mirror key + resolved HEAD `sha-<12hex>` version; the pinned per-sha path is byte-unchanged.**

## Accomplishments

- Forked `makeInstallCloneProbe` and `makeUpdateCloneProbe` on `source.sha === undefined`: an unpinned source (incl. ref-only moving pointers) routes to the mirror seam instead of the per-sha immutable cache. The pinned arm still runs `resolvePluginPin` → `materializePluginClone` unchanged.
- Extended `InstallCloneCacheSeam` and `UpdateCloneCacheSeam` with `materializeOrRefreshPluginMirror` (required member) plus the default injections; the fork lives entirely inside the probe callback so the orchestrator bodies never name the git surface.
- Unpinned install records the bare `plugin-clones/<12hex>/` mirror root (+ subdir for git-subdir) and the `sha-<12hex>` of the mirror HEAD; unpinned update refreshes the mirror in place and re-anchors the record from its old per-sha key to the bare mirror key, GC'ing the now-unreferenced per-sha clone (feeds MIRR-04 in Plan 04).
- The `no-orchestrator-network` gate stays green: install.ts names no git surface; update.ts remains the only sanctioned gitOps-exempt orchestrator (no NEW forbidden target introduced).

## Decisions Made

- **Required, not optional, seam member.** The plan says "extend the interface with a new member"; making it required preserves the type safety the phase depends on. This forced additive updates to every seam-builder helper in the consumer test files (Rule 3 blocking, scoped strictly to the interface change).
- **Extracted shared helpers to hold the complexity gate.** Adding the fork pushed `makeInstallCloneProbe` over the `sonarjs/cognitive-complexity` limit (17 > 15). Rather than suppress the rule, I extracted `buildProbeAuth` + `resolveGitPluginRootWithSubdir` (install) and `buildBundle` (update), which also removes the pre-existing auth-bundle/subdir-tail duplication between the two arms.
- **Vanished-repo failure now comes from the mirror clone.** The old unpinned failure test drove `resolveRemoteRefThrows`; the mirror route never resolves a remote ref, so the equivalent network failure is the cold-clone throw (`cloneThrows`), which propagates through the probe and classifies into the existing `{network unreachable}` REASON token unchanged.
