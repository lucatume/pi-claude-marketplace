<objective>
Refine the bulk `update` cascade + summary grammar (UGRM-01, UGRM-02). A bulk
`update` must stop rendering a `(skipped) {up-to-date}` row for every unchanged
plugin (UGRM-01), and its headline must count realized transitions only —
`Plugin update: N updated` instead of `N successes`, which today inflates the
count with at-desired-state info rows (UGRM-02). A bulk update that realized ZERO
transitions (all up-to-date, or only info skips such as a `(force-upgradable)`
decline) must still emit a single non-silent headline — never zero output and
never a vanished summary line.

The change is UPDATE-SCOPED. The shared `composeTally` helper is consumed by
install / reinstall / marketplace / import; the success-count fix is delivered
as an OPT-IN optional `tally` override on the cascade envelope so every other
operation's summary stays byte-identical. The single-target `update
<plugin>@<mp>` path is NOT suppressed (a user who named one plugin still sees
its up-to-date skip row).

Purpose: align the bulk-update grammar with the project's own UXG-02 benign-no-op
suppression philosophy and make the headline read as operations-performed.
Output: the new optional `tally` field on `CascadeNotificationMessage`, the
orchestrator suppression + no-op + stamping logic, and lockstep docs/test byte
relocks. `npm run check` green is the phase gate.
</objective>

<plan_decisions>
These strings are LOCKED by this plan and become the byte contract in
`docs/output-catalog.md`. Do not paraphrase them.

1. **Zero-realized-transition no-op line (covers BOTH the all-up-to-date case
   AND the non-empty-cascade-but-0-updated case):**
   `Plugin update: nothing to update` (info severity, no reload-hint). Chosen
   over `Plugin update: 0 updated` — shorter, matches the UXG-02 "benign no-op
   gets one clear line" philosophy and the hard-coded `reconcile-pending-empty`
   precedent (`notify.ts:3181-3186`).

   The orchestrator emits this single headline whenever the bulk update realized
   ZERO transitions AND has zero failures and zero warnings — i.e.
   `updatedCount === 0` AND no `error`/`warning` rows survive — regardless of
   whether the post-suppression cascade is empty (all up-to-date) or still has
   info rows (e.g. the post-Phase-73 `skip-force-upgradable-bulk`
   `(force-upgradable)` row). This is the fix for the byte-drift defect: stamping
   `tally {count: 0}` makes `composeTally` return `""`, so the orchestrator must
   own the never-silent headline rather than letting the override swallow it.

   - **Empty cascade (all up-to-date):** ONLY the `Plugin update: nothing to
     update` line is emitted (no cascade body) — see `all-up-to-date-noop`.
   - **Non-empty cascade, 0 updated, 0 failures, 0 warnings:** the cascade body
     (the surviving info rows, e.g. the `(force-upgradable)` row + its trailer)
     IS rendered, followed by `Plugin update: nothing to update` as the tally
     line — see `skip-force-upgradable-bulk`. The body still flows through
     `notifyWithContext`; only the headline is the no-op constant.

2. **Updated headline verb:** `Plugin update: N updated` — `"updated"` is BOTH
   the singular and plural form (`1 updated`, `3 updated`); there is no
   plural-s. Reuse `tallyCategory(count, "updated", "updated")` (do NOT pass
   `"updateds"`).

3. **Mixed failure+updated headline — the override COMPOSES, it does not
   replace the whole line.** The `tally` override owns ONLY the success
   category (the count is realized-transition rows, rendered with the verb
   `updated`); the failure and warning categories are still computed from the
   existing `countRowsBySeverity` math and fold in ahead of it in the existing
   `failures, warnings, success` order. So after suppressing the up-to-date
   `beta` row, `single-mp-mixed` becomes `Plugin update: 1 failure, 1 updated`
   and `bare-multi-mp` becomes `Plugin update: 1 failure, 2 updated`. The
   `failed-with-rollback-partial` state (plural, one `failed` row, NO info rows)
   is INTENTIONALLY UNCHANGED at `Plugin update: 1 failure` — there are zero
   `updated` rows so the override contributes nothing, and the failure category
   is unchanged math.

4. **The updated count predicate:** `outcomes.filter(o => o.outcome.partition
   === "updated").length`. Force-installed degraded updates live INSIDE the
   `updated` partition (`update.ts:1619-1646`), so they are counted. This is
   derived BEFORE row suppression, independent of UGRM-01 filtering. The
   `force-upgradable` DECLINE row is partition `skipped` (Phase 73), NOT
   `updated`, so it correctly contributes 0 to the count.
</plan_decisions>
