<objective>
Replace the resolver's binary `installable: true | false` discriminated union with
a three-way `state: "installable" | "unsupported" | "unavailable"` union, split the
single `dirty` accumulator into a structural-defect signal vs an unsupported-component
signal with structural precedence (D-64-07), add the `requireForceInstallable` narrowing
gate beside `requireInstallable`, and re-point every resolver-union consumer to `r.state`.

Purpose: Establish the type-level foundation so the milestone can later distinguish
"force can degrade the unsupported parts" (`unsupported`) from "force cannot help"
(`unavailable`), with NFR-7 refined (not weakened) — reading `pluginRoot` off a
structurally-broken plugin becomes a compile error.

Output: A three-arm union + two gates in `domain/resolver.ts`, migrated consumers
(`list`, `info`, `edge-deps`), and migrated/extended resolver tests. `npm run check`
is green at the plan boundary.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| on-disk plugin manifest/hooks → resolver | Untrusted plugin content (manifest, hooks.json, component paths) crosses into the resolver, which classifies it structurally before any consumer reads a filesystem path off it. |
| resolver value → consumer (force gate) | The discriminated `state` tag is the only thing standing between a structurally-broken plugin and a consumer that would read its `pluginRoot` / materialize components. |

A structurally-broken plugin's filesystem root never reaches a consumer. |
| T-64-02 | EoP | `requireForceInstallable` gate | mitigate | The gate's assertion target is `ResolvedPluginInstallable \| ResolvedPluginUnsupported` only; it throws on `unavailable`. Covered by gate tests (Task 3). Force can never (in Phase 65) bypass a structural/NFR-10 failure (FORCE-05 forward-looking). |
| T-64-03 | EoP | NFR-10 path containment / source escape | mitigate | `preflightStages` (`sourceEscapeReason`) and `validateComponentPath` failures are STRUCTURAL → route to `unavailable` via the structural accumulator (Task 1), strengthening — not weakening — containment vs the binary form. |

| T-64-SC | Tampering | npm/pip/cargo installs | accept | No packages installed this phase (pure in-repo TypeScript refactor; RESEARCH "Package Legitimacy Audit: Not applicable"). No supply-chain surface introduced. |

No HIGH-severity unmitigated threats — this phase REFINES the NFR-7 / NFR-10 compile-time guarantees rather than weakening them.
