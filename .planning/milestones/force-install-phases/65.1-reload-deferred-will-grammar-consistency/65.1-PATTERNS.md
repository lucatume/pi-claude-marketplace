## Orienting principle (read first)

This is a token-**removal** refactor: it retires the two marketplace-level
pending tokens `"will add"` / `"will remove"` and decrements the closed-set
tripwires. It is the exact INVERSE of the prior token-ADDING phases (DIFF-02,
ENBL-04) that introduced the `will *` family.

For a removal, the closest analog is NOT a deleted token (none exist to copy) —
it is the **structure of a surviving sibling token**. The four plugin-level
`will install` / `will uninstall` / `will enable` / `will disable` tokens SURVIVE
this phase and appear at every one of the same edit sites (tuple entry, type
interface, union arm, render arm, projection loop, render-map arm, tripwire,
fixture, doc row). Each surviving sibling shows two things at once:

1. The exact shape of what is being deleted (the retired `will add`/`will
   remove` arms were modeled on these), and
2. The shape that must remain intact after deletion (so the planner does not
   over-cut).

Mirror the sibling's structure in reverse: wherever `will add`/`will remove`
sit two lines above/below a surviving `will install`/`will uninstall`, delete
ONLY the marketplace pair and leave the plugin siblings byte-identical.

**Comment/test-title policy (`.claude/rules/typescript-comments.md`):** use
`D-65.1-NN` / `WILL-NN` / `NFR-N` / `SNM-NN` / `DIFF-NN` IDs; NEVER GSD
phase/plan references. The new WILL-02 test title must use `WILL-02`, not a
phase number.

**Atomic-supersession constraint (D-65.1-04 / Pitfall 2):** all five enforcement
surfaces (closed-set tuples, render arms, tripwire counts, catalog-uat fixtures,
prose docs) change in ONE commit. `npm run check` must be green at the commit
boundary, never just between commits.

## Shared Patterns

### Closed-set removal protocol (applies to every code+test+doc site)
**Source of truth:** `shared/notify.ts` tuples + `assertNever` tails.
**Apply to:** notify.ts, reconcile/notify.ts, all closed-set tests, both docs.
A token retirement touches, in lockstep: (1) the runtime tuple entry, (2) the
type interface + union arm, (3) the render switch arm, (4) the length tripwire
literal, (5) the catalog-uat fixture, (6) the prose doc row/count. The
`assertNever` in `renderMpHeader` and the `Extract` in `ReconcileBlockStatus`
are the compile-time tripwires that will go red if any of (2)/(3) is missed —
let them guide the deletion. The runtime `assert.equal(...length, N)` tripwires
catch (1). This is the inverse of the token-add protocol that introduced these
same arms.

### Bare list-arm header reuse (no new vocabulary)
**Source:** `renderMpHeader` `case undefined:` (notify.ts:1506-1513) +
`renderWillUninstall` (reconcile.messaging.ts:87-88).
**Apply to:** the removed-marketplace cascade and the child-install-bearing added
marketplace. Both retired marketplace tokens are replaced by the EXISTING
status-less header + existing per-plugin `will *` child rows. No render arm, no
token, no icon constant is added. The catalog's `plugin-pending-uninstall` state
(output-catalog.md:1343-1352) is the canonical byte form to match.

### Pure-projection seam (Pitfall 1)
**Source:** `PlannedMarketplaceRemove` (types.ts:63-66) populated in
`diffMarketplaces` (plan.ts:210-216).
**Apply to:** the removed-marketplace plugin-name source. Carry
`readonly plugins: readonly string[]` on the plan DTO rather than threading
`state` into the projection — keeps `buildReconcilePendingNotification` pure and
the planner-purity test green. Do NOT enumerate removed-marketplace plugins into
`pluginsToUninstall` (would double-bill the apply cascade, plan.ts:366-387).
