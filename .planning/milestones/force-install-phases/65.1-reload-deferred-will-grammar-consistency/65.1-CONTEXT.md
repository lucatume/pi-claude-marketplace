# Phase 65.1: Reload-Deferred Will Grammar Consistency - Context

**Gathered:** 2026-06-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Make the `will` prefix in the pending/preview (`/claude:plugin pending`) surface
mean exactly one thing: the action's effect is deferred to the next `/reload`.
Actions that take effect immediately render without a `will` token (and, when
they have no reload-deferred consequence, do not appear in the pending preview
at all). This reconciles the pending-tense grammar with the per-command
reload-hint discipline BEFORE the force-display phases (66+) add their own
`will force …` tokens on top.

This is a consistency/correctness refactor of the EXISTING (already-shipped)
reconcile grammar — discovered mid-milestone while resolving FSTAT-06's
`will force update` (which has no surface because reconciliation has no update
action). Inserted as Phase 65.1 so Phase 66's force-preview tokens build on the
corrected grammar.

Requirements WILL-01..04 are locked by the ROADMAP; the EXACT per-action
reload-deferral facts are a research deliverable (see Research Questions).

</domain>

<decisions>
## Implementation Decisions

### Authoritative classifier (WILL-02)
- **D-65.1-01:** The per-command reload-hint discipline — the
  `/reload to pick up changes` trailer (governed by existing decisions
  D-17.1-01 / D-18-04 / D-03 / D-06) — is the SINGLE source of truth for "is
  this action reload-deferred?". The pending `will`-grammar must be consistent
  with it: a pending row carries `will` exactly when its corresponding command
  path emits the reload-hint trailer. No independent reload-deferral logic in
  the pending surface. WILL-02 is the verification anchor (a test asserting the
  two surfaces agree).

### Immediate-action rendering (WILL-01)
- **D-65.1-02:** An action that is immediately effective is DROPPED from the
  pending preview entirely — `pending` answers "what will the next reload do?",
  so an immediate action has nothing pending. A change consisting only of
  immediate actions yields the existing empty form
  (`Pending: next reload will apply 0 actions.`). No new present-tense tokens
  are added for immediate actions.

### Marketplace remove split (WILL-03)
- **D-65.1-03:** Marketplace de-registration is immediate (NO `will remove`
  marketplace token). Removing a marketplace that still has installed plugins
  renders its reload-deferred work as per-plugin `will uninstall` rows in
  pending; a marketplace with no installed plugins shows nothing pending. This
  RETIRES the marketplace-level `will add` / `will remove` pending tokens from
  the closed status set (the inverse direction from the token-adding phases —
  the closed-set length tripwires DECREMENT). Plugin install / uninstall /
  enable / disable remain reload-deferred and keep their `will` tokens.

### Byte-contract scope (WILL-04)
- **D-65.1-04:** Update EVERYTHING in this phase: the notify tests, the
  catalog-uat byte-equality runner, AND the prose docs
  (`docs/output-catalog.md`, `docs/messaging-style-guide.md`). Self-contained —
  the reconciled grammar is fully documented here. Phase 70 only performs the
  final PRD §11 reconcile, not these mid-milestone byte changes. `npm run check`
  must stay green.

### Claude's Discretion
- Helper naming and where the reload-hint→will-token consistency seam lives
  (shared classifier vs assertion), provided behavior matches D-65.1-01..04.
- Whether the marketplace reconcile rows are removed from the pending render
  map (MOD-03) entirely or short-circuited upstream — planning's call.

</decisions>

4. **Pending reconcile render path.** Map how `orchestrators/reconcile/pending.ts`
   + `reconcile.messaging.ts` (MOD-03 render map) currently emit marketplace

<specifics>
## Specific Ideas

Origin: surfaced while resolving FSTAT-06's `will force update`. There is no
`will update` because reconciliation never bumps versions — it only makes state
match config (install/uninstall/enable/disable for plugins; add/remove for
marketplaces). The deeper inconsistency the user spotted: marketplace add is
immediate yet renders `will add`, and marketplace remove is immediate except
for its reload-deferred plugin-uninstall cascade. This phase makes `will`
strictly mean "happens on the next reload."

</specifics>
