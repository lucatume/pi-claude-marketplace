# Phase 65.1: Code Review Report

**Depth:** standard

## Summary

Reviewed the retirement of the two marketplace-level pending tokens (`will add` / `will remove`) so that `will` means exactly "deferred to the next /reload". The five focus areas were checked directly:

1. **Projection/render logic (reconcile/notify.ts).** Correct. The `marketplacesToRemove` per-plugin cascade (`pushMarketplaceRemoveCascade`) synthesizes one `will uninstall` child row per recorded plugin under a bare list-arm header (status `undefined`), and skips the block entirely when `o.plugins.length === 0`. The dropped `marketplacesToAdd` loop is genuinely gone (only a comment remains). `ReconcileBlockStatus` narrowing and the shared `blockToMarketplaceMessage` switch ("added"/"removed" arms stay live via the applied-cascade path) are sound — no dead arm.

2. **The new `PlannedMarketplaceRemove.plugins` seam.** Correct and non-double-billing. `diffMarketplaces` populates it from `Object.keys(mpRecord.plugins)`; `buildUninstallBucket` continues to skip removed-marketplace plugins; and crucially `applyMarketplaceRemoves` (apply.ts:314-358) reads only `op.marketplace`/`op.scope` and never `op.plugins`, so the field is consumed by the pending projection alone. `plan.test.ts` and `apply.test.ts` both assert `pluginsToUninstall.length === 0` for removed marketplaces.

3. **Retired-token / dead-code sweep.** Two stale references remain (WR-01, WR-02). No live code path still emits or consumes `will add` / `will remove`.

4. **Empty-plugins guard + `isReconcilePlanListEmpty`.** Consistent with the projection: a remove contributes a pending row iff it carries plugins, `marketplacesToAdd` is no longer counted, and the empty-vs-non-empty decision matches `buildReconcilePendingNotification` exactly (verified by case analysis). One by-design semantic edge noted as IN-01.

5. **Byte-contract integrity.** The closed-set locks (STATUS_TOKENS 20, MARKETPLACE_STATUSES 7, PLUGIN_STATUSES 15, REASONS 32) match the tuples; the new `marketplace-remove-with-installed-plugins` catalog fixture is byte-identical to its catalog block and is covered by the bidirectional catalog-UAT walk.

No correctness, security, or data-loss defects found. The two warnings are stale comments that misstate the post-65.1 grammar (a maintenance/traceability hazard, not a behavior bug).

## Warnings

### WR-01: Stale "6 will*" comment after STATUS_TOKENS decrement

**File:** `extensions/pi-claude-marketplace/shared/notify.ts:188`
**Issue:** The DIFF-02 doc block over `STATUS_TOKENS` still reads "the 6 `"will *"` entries are the pending-tense tokens". After this phase retired `will add` / `will remove`, `STATUS_TOKENS` carries only **four** `will *` entries (`will install`, `will uninstall`, `will enable`, `will disable`) — the lock test (`notify-closed-set-locks.test.ts`) now pins the tuple at 20. The "6" is a retired-token reference that survived the decrement and now contradicts the very tuple it annotates. (The adjacent "four head-of-tuple state-change tokens" clause is still accurate.)
**Fix:**
```ts
 * DIFF-02 (D-53-02): the 4 `"will *"` entries are the pending-tense tokens
 * emitted by `/claude:plugin pending` rows. ...
```

### WR-02: Stale `will add` / `will remove` marketplace-header description

**File:** `extensions/pi-claude-marketplace/orchestrators/reconcile/reconcile.messaging.ts:40`
**Issue:** The module doc states: "The marketplace header (`will add` / `will remove` for pending; `added` / `removed` for applied; `failed` for both) ... stay central in `notify.ts`". Post-65.1 the pending marketplace header NEVER carries `will add` / `will remove` — those tokens are retired; a pending marketplace header is now always bare (status `undefined`) or `failed` (source mismatch). This comment documents the retired behavior and will mislead a future maintainer of this module about the pending grammar. (This file is outside the explicit review file list but is directly invalidated by the phase change and is the same retired-token class as WR-01.)
**Fix:** Drop the `will add` / `will remove` clause; describe the pending header as bare/list-arm (status-less) or `failed`, e.g.:
```ts
 * (`failed` for source-mismatch pending; `added` / `removed` for applied;
 * a bare status-less list-arm header otherwise), the cause-chain / rollback ...
```

## Info

### IN-01: `pending` reports "0 actions" while a plugin-less marketplace removal still de-registers on next reload

**File:** `extensions/pi-claude-marketplace/orchestrators/reconcile/notify.ts:330-339`
**Issue:** `isReconcilePlanListEmpty` treats a `marketplacesToRemove` entry with no recorded plugins as non-contributing, so a plan whose only action is an immediate marketplace de-registration routes to the `Pending: next reload will apply 0 actions.` advisory. The next reload will in fact mutate `state.json` (de-register the marketplace). This is an explicit, documented design choice (D-65.1-03 — "a removal with no recorded plugins is immediate de-registration and contributes nothing"), aligned with the marketplace-records-are-bookkeeping / reload-hint-suppression philosophy, and is locked by the `WILL-03: MarketplaceRemove with no recorded plugins -> no pending block` test. Flagged only for explicit confirmation that "0 actions" is the intended user-facing wording for the immediate-de-registration-only case.
**Fix:** No change required if intended. If the literal "0 actions" is a concern, the advisory could distinguish "0 reload-deferred actions" — but that would change the catalog byte contract, so confirm intent before touching it.

### IN-02: Pre-existing style-guide drift (`present` / "16" counts) in an edited doc

**File:** `docs/messaging-style-guide.md:26,33,44`
**Issue:** The guide still says `PluginStatus; // 16 literal strings`, "the union has sixteen variants", and lists `PluginPresentMessage // status: "present"`. `PLUGIN_STATUSES` is now 15 and `present` was collapsed into `installed` long before this phase, so these are stale. This drift is **pre-existing** and unrelated to the `will add`/`will remove` retirement, but `messaging-style-guide.md` was edited in this phase (the MARKETPLACE_STATUSES "7" line was correctly updated), making it a low-cost opportunity to also correct the adjacent counts.
**Fix:** Update the count to 15, drop the `PluginPresentMessage` row, and reword "sixteen variants" to "fifteen variants". (Out of strict phase scope — fix only if touching the file.)
