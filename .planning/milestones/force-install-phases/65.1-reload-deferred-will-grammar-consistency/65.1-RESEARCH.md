# Phase 65.1: Reload-Deferred Will Grammar Consistency - Research

**Researched:** 2026-06-27
**Domain:** Internal byte-exact notification-grammar refactor (TypeScript closed-set unions, reconcile pending-preview renderer)
**Confidence:** HIGH (every claim below is traced to a specific file:line in the live codebase)

## Summary

This is a code-tracing refactor, not a library/stack problem. The authoritative
reload-deferral oracle already exists and is mechanically inspectable:
`shouldEmitReloadHint` in `shared/notify.ts` (line 2465) is the OR-reduce of the
caller-stamped per-row `needsReload` flag over the flattened marketplace + plugin
rows. The trailer `/reload to pick up changes` (literal at line 2043) fires iff at
least one row stamps `needsReload: true`. There is NO status-token or
cascade-kind inference any more — the v1.x notification refactor (commit 49325575)
replaced that with per-row stamps (JSDoc lines 2440-2463 spell out the exact
trigger set). This means D-65.1-01's "single source of truth" already physically
exists; the pending `will`-grammar simply has to align to it.

Tracing each command's success-row construction against `needsReload` produces an
unambiguous classifier (full table below). The headline facts: **marketplace add
stamps no `needsReload` on its `(added)` row → never reload-deferred → must lose
`will add`. Marketplace remove stamps `needsReload: false` on the `(removed)`
header but `needsReload: true` on each per-plugin `(uninstalled)` child → the
de-registration is immediate, only the plugin-uninstall cascade is reload-deferred
→ `will remove` is retired and the cascade renders as per-plugin `will uninstall`
rows. Plugin install / uninstall / enable / disable all stamp `needsReload: true`
on their realized transitions → reload-deferred → keep their `will *` tokens.**

The closed-set impact is exact and small: retire `"will add"` and `"will remove"`
from `STATUS_TOKENS` (22→20) and `MARKETPLACE_STATUSES` (9→7); `PLUGIN_STATUSES`
(15) and `REASONS` (32) are untouched. The pending render path lives in
`orchestrators/reconcile/notify.ts` (`buildReconcilePendingNotification`) and
`reconcile.messaging.ts` (`PENDING_CONTEXT` plugin-row map); the marketplace
header arms live centrally in `renderMpHeader` (notify.ts 1494/1499).

**Primary recommendation:** Delete the `marketplacesToAdd` projection loop
(immediate; nothing pending unless it carries child installs, which already get
their own `will install` rows under a bare list-arm header). Replace the
`marketplacesToRemove` loop so that, instead of emitting a `will remove`
marketplace header, it emits one `will uninstall` child row per recorded plugin
under that marketplace (the recorded plugin names live in `state`, not in the
plan today — see Pitfall 1). Retire the two marketplace `will *` tokens, types,
render arms, tripwire counts, doc rows, and tests in lockstep.

## Research Question 1 — Per-action reload-deferral ground truth (DEFINITIVE TABLE)

**The classifier (D-65.1-01 oracle):** `shouldEmitReloadHint(message)` —
`extensions/pi-claude-marketplace/shared/notify.ts:2465-2521`. It returns
`true` iff `mp.needsReload === true` for any marketplace row OR
`p.needsReload === true` for any plugin row (lines 2508-2518). Info-surface kinds
(incl. `reconcile-pending-empty` and `reconcile-applied-cascade`) short-circuit to
`false` (lines 2476-2499). The JSDoc at 2440-2463 states the canonical rule:
"every marketplace status (added / removed / updated / autoupdate enabled /
autoupdate disabled / skipped / failed) stamp `needsReload: false`"; only realized
**plugin** install/update/reinstall/uninstall and the fresh-disable transition
stamp `true`. The trailer literal is `RELOAD_HINT_TRAILER` (line 2043).

| Action | Emits reload-hint trailer? | Reload-deferred or Immediate | Keeps `will` token? | Evidence (file:line) |
|--------|---------------------------|------------------------------|---------------------|----------------------|
| **marketplace add** | NO — `(added)` row has no `needsReload` stamp (defaults false); registering a marketplace record is bookkeeping, not a Pi-visible resource | **Immediate** | **DROP `will add`** | add.ts:569-576 (added row, no `needsReload`); shouldEmitReloadHint JSDoc notify.ts:2440-2443, 2450-2452 |
| **marketplace remove — source de-registration** (empty remove, no plugins) | NO — `(removed)` header stamps no `needsReload`; empty `plugins: []` → OR-reduce false (header-only, G-MIL-02) | **Immediate** | **DROP `will remove`** | remove.ts:761-777 (removed header, no `needsReload`); comment remove.ts:752-757; shouldEmitReloadHint notify.ts:2508-2520 |
| **marketplace remove — plugin-uninstall cascade** (remove with installed plugins) | YES — each child `(uninstalled)` row stamps `needsReload: true` | **Reload-deferred** (the cascade only) | **Render as per-plugin `will uninstall` rows** (no marketplace token) | remove.ts:766-774 (`needsReload: true` per unstaged plugin); D-22-01/D-22-02 |
| **plugin install** | YES — `(installed)` realized row stamps `needsReload: true` | **Reload-deferred** | **Keep `will install`** | install.ts:1399 (`needsReload: true`) |
| **plugin uninstall** | YES — `(uninstalled)` realized row stamps `needsReload: true` | **Reload-deferred** | **Keep `will uninstall`** | uninstall.ts:650 (`needsReload: true`) |
| **plugin enable** | YES — re-materializes via install; fresh `(installed)` row stamps `needsReload: true` | **Reload-deferred** | **Keep `will enable`** | enable-disable.ts:1001 (`needsReload: true`); reconcile apply mirror notify.ts:377-386 |
| **plugin disable** | YES — fresh `(disabled)` transition stamps `needsReload: true` (inventory `(disabled)` stamps false) | **Reload-deferred** | **Keep `will disable`** | enable-disable.ts:1014 (`needsReload: true`); comment 852-856 |

Supporting cross-checks (not pending-surface actions but confirm the oracle is
uniform): plugin update `(updated)` stamps `needsReload: true` (update.ts:1573);
reinstall stamps `needsReload: true` (reinstall.ts:313, 879). There is no
reconciliation `will update` because reconcile never bumps versions (CONTEXT
`<specifics>`), which is why FSTAT-06's `will force update` has no reconcile
surface — the discovery that triggered this phase.

## Research Question 2 — Is marketplace add ever reload-deferred?

**No.** A config-declared marketplace is reconciled at load by `applyReconcile`
(the apply path runs ON `/reload`), and the realized `(added)` marketplace header
is dispatched through the `reconcile-applied-cascade` kind, which
`shouldEmitReloadHint` short-circuits to `false` structurally (notify.ts:2487-2494)
AND whose header stamps no `needsReload` anyway. The standalone `marketplace add`
command likewise emits no trailer (add.ts success path stamps nothing; the
JSDoc comment at add.ts:31-34 that says "mp.status `added` is state-changing" is
STALE prose from before the per-row-stamp refactor — the live `shouldEmitReloadHint`
no longer inspects `mp.status`, only `mp.needsReload`). Therefore a pending
marketplace add is, by the D-65.1-01 oracle, NOT reload-deferred and must not
carry `will add`. **D-65.1-02 and D-65.1-03 validated.**

Nuance the planner must preserve: a newly-added marketplace that ALSO has child
plugins to install still appears in the pending preview — but as a bare list-arm
container header (`● new-mp [user]`, no token) whose children carry `will install`.
A marketplace add with NO child installs must produce NO block at all (D-65.1-02
"dropped entirely"). The existing catalog `plugin-pending-uninstall` state already
demonstrates the exact bare-header byte form (output-catalog.md:1349-1352).

## Pitfalls

### Pitfall 1: The planner deliberately omits per-plugin uninstalls for a removed marketplace

**What goes wrong:** Naively dropping `will remove` and expecting `pluginsToUninstall`
to already contain the removed marketplace's plugins. It does NOT.
**Why it happens:** `buildUninstallBucket` (plan.ts:366-387) explicitly `continue`s
past any marketplace not in `merged.marketplaces` and not in `declaredAndRecorded`
— i.e. a marketplace in `marketplacesToRemove` contributes ZERO `pluginsToUninstall`
entries (comment plan.ts:359-365: "torn down whole-cloth by the apply path; listing
each plugin under it as a separate uninstall would double-bill the work"). The apply
path relies on this (removeMarketplace cascades internally), so the planner must
NOT be changed to enumerate them (it would double-bill the apply cascade).
**How to avoid:** Synthesize the `will uninstall` child rows in the PENDING path
only, from `state.marketplaces[mpName].plugins` (the recorded plugin names).
`pendingReconcile` already holds the per-scope `state` (pending.ts:163). Two clean
options (Claude's discretion D-65.1-03): (a) carry `readonly plugins: readonly
string[]` on `PlannedMarketplaceRemove` (types.ts:63-66), populated in
`diffMarketplaces` (plan.ts:210-217) which has `state` in scope — keeps the
projection pure; or (b) thread per-scope `state` into
`buildReconcilePendingNotification`. Option (a) is type-cleaner and keeps the
purity gate (`reconcile-planner-purity.test.ts`) and projection-purity intact.
**Warning sign:** A `marketplace remove` of a populated marketplace shows an empty
pending preview, or `npm run check` passes but the catalog has no
removed-marketplace-cascade state.

### Pitfall 2: Byte-exact contract — every token edit needs lockstep test + doc updates

**What goes wrong:** Changing the closed set without updating the catalog UAT and
docs in the SAME commit → `npm run check` goes red between commits (CLAUDE.md and
the v1.3 atomic-supersession lesson forbid this).
**Why it happens:** Five enforcement surfaces lock the bytes simultaneously.
**How to avoid:** Treat the following as ONE atomic change set (full map):
- `tests/architecture/notify-closed-set-locks.test.ts` (22→20, 9→7).
- `tests/architecture/catalog-uat.test.ts:2905-2916` — `mp-add-plugin-install`
  fixture currently sets `status: "will add"` (line 2912); drop the status field
  (becomes bare list-arm header). Add a new catalog state for
  marketplace-remove-with-installed-plugins → per-plugin `will uninstall` rows.
- `tests/architecture/notify-grammar-invariant.test.ts:172-182` — delete the two
  `will add` / `will remove` marketplace-header fixtures; update the row-grammar
  regex (line 231) to drop `add|remove` from the `(will (?:...))` alternation.
- `tests/shared/notify-v2.test.ts:3986-4111` — remove/rewrite the will-add /
  will-remove `renderMpHeader` render assertions (e.g. line 4015 asserts
  `○ old-mp [project] (will remove)`).
- `tests/orchestrators/reconcile/notify.test.ts:57,120,128-138` — projection
  tests asserting `block.status === "will add"` / `"will remove"`.
- `tests/orchestrators/reconcile/apply.test.ts:1132` — regex asserting

- `tests/orchestrators/reconcile/pending.test.ts:453-454` — asserts a local-only

- `docs/output-catalog.md` — glyph legend lines 9-10 (remove `will add`/`will
  remove` mentions), token table rows 156-157 (DELETE both), the disabled-row
  cross-reference at line 315, the pending intro at line 1320 (drop `will add`/`will
  remove` from the token list), and the `Marketplace add with child plugin install`

- `docs/messaging-style-guide.md:27,58` — count 9→7 and remove the `will add`/`will

### Pitfall 3: Stale prose that LOOKS authoritative

**What goes wrong:** Trusting in-code comments over the live `needsReload` logic.
**Why it happens:** `add.ts:31-34` claims the trailer is "computed by notify()
(mp.status `added` is state-changing)" — false under the current per-row-stamp
model. `docs/output-catalog.md:1368` says "REASONS stays at 29 entries" while the
tripwire asserts 32. These are pre-existing drift, not load-bearing.
**How to avoid:** The only authority for reload-deferral is
`shouldEmitReloadHint` + the per-row `needsReload` stamps. Fix the `add.ts:31-34`
comment opportunistically if the planner touches add.ts; the REASONS-29 prose is
out of scope (REASONS untouched) but worth a one-line correction since docs are
being edited anyway (D-65.1-04).

// Source pattern: extensions/pi-claude-marketplace/orchestrators/reconcile/notify.ts:219-222
// BEFORE:
