<objective>
Gate the install/update tab-completion candidate sets on a `--force` token
preceding the plugin positional, sourcing the sets from the finer cache statuses
produced in 67-03 (LIST-02 / D-67-02).

Purpose: with `--force` present, `install` completion should offer the
force-installable candidates (`available` + `unsupported`) and `update`
completion should offer the force-upgrade candidates (`upgradable` +
`force-upgradable`); `unavailable` is excluded in both. Without `--force`,
completion output must be byte-identical to today — install offers only
`available`, and update offers ALL installed-family statuses (which, before the
67-03 split, were a single `installed` bucket). `--force` is also surfaced as a
flag completion for install/update (and is no longer offered for reinstall after
67-01), and registering it as a boolean flag for positional extraction fixes the
`install --force <TAB>` → null bug.

Output: provider `--force` flag entry + presence detection threaded into the
plugin-ref branch; `data.ts` per-(mode, force) candidate filters over the finer
`PluginIndexRow` statuses; new provider tests for the force candidate sets plus a
no-force byte-identical regression; full `npm run check` green.
</objective>

No NFR-5 surface: the completion path does no network I/O; candidate statuses come from the no-network cache. No render-byte/catalog change. No HIGH-severity threats.
