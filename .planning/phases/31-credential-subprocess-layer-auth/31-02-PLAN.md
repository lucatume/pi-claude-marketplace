<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| extension -> OS keychain | Mediated by `git credential` subprocess; no direct keychain API call from extension code |
| extension -> spawned `git` binary | argv contains only fixed subcommand names ("credential", "fill"|"approve"|"reject"); no user input enters argv (input is fed over stdin per git-credential wire format) |
| platform/git-credential.ts -> caller (Phase 32+) | Returns GitCredentials | null on fill; void on approve/reject. Caller must not log the returned credential without scrubbing |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-31-02-01 | Information Disclosure | Error messages from gitCredentialIO | mitigate | Error messages reference operation name + exit code ONLY (per RESEARCH Pitfall 8); no template literal interpolates a password / cred field. Asserted by tests/architecture/no-credential-leak.test.ts second test. |
| T-31-02-02 | Information Disclosure | Credentials in spawn argv | mitigate | Credentials flow exclusively over stdin (per RESEARCH "Anti-Patterns to Avoid" item 1 -- argv visible via /proc/<pid>/cmdline / ps -ef). argv contains only `"credential"` + `"fill"|"approve"|"reject"`. |
| T-31-02-03 | Denial of Service | Hung subprocess waiting for stdin EOF | mitigate | Both `child.stdin.write(input)` AND `child.stdin.end()` are called per RESEARCH Pitfall 3 + Pattern 1; a 5_000ms setTimeout with `.unref()` kills the child via SIGTERM on timeout (RESEARCH Pattern 1 CP-4). |
| T-31-02-04 | Denial of Service | Interactive TTY prompt deadlock | mitigate | `env: { ...process.env, GIT_TERMINAL_PROMPT: "0" }` per RESEARCH Pitfall 2 -- prevents git from falling through to the TTY prompt on a credential helper miss. |

| T-31-02-06 | Elevation of Privilege | OS-keychain helper executing under extension UID | accept | The helper runs under the user's UID (same as the Pi process) -- not a privilege boundary. NFR-10 containment unaffected (helper writes to system-managed store, not to scopeRoot). |
| T-31-02-07 | Tampering | Whitelist file path drift | mitigate | Plan 31-01's ALLOWED_CHILD_PROCESS_FILES already pins the exact path; this plan creates the file at that exact path so the whitelist holds. |
| T-31-02-SC | Tampering | npm/pip/cargo installs | mitigate | No new dependencies (RESEARCH "Installation: None"). `node:child_process` and `node:test` are Node built-ins; `isomorphic-git` is an existing dep (used only via its re-exported `GitCredentials` type from platform/git.ts). |

    Internal helper `gitCredentialIO(subcommand, input, timeoutMs = 5_000)`:
    - Spawns `git credential <subcommand>` via `spawn` from `node:child_process` with `stdio: ["pipe", "pipe", "pipe"]` and `env: { ...process.env, GIT_TERMINAL_PROMPT: "0" }`.
    - Buffers stdout and stderr; clears the timeout on `close`; the timeout (default 5_000ms) sends SIGTERM and rejects with `git credential <subcommand> timed out after <ms>ms`. The timeout handle MUST call `.unref()` so a pending timer cannot keep the host Pi process alive past completion.
    - Resolves with `{ stdout, stderr, code }` (code defaults to -1 if null per RESEARCH Pattern 1).
    - Calls `child.stdin.write(input)` followed immediately by `child.stdin.end()` (both required per RESEARCH Pitfall 3).
    - Handles `child.on("error", err => reject(err))` to surface ENOENT correctly.

    `credentialFill(host)` behavior (AUTH-08):
    - Builds input via `buildAttributeBlock(host)` -- output: `"protocol=https\nhost=<host>\n\n"` (NO username, NO password, NO path per RESEARCH Pitfall 4).
    - Wraps `gitCredentialIO("fill", input)` in try/catch; on ENOENT / timeout / any throw, return `null` (RESEARCH Pitfall 7).
    - On `result.code !== 0`, return `null` (the GIT_TERMINAL_PROMPT=0 miss path; RESEARCH Pattern 2 + Pitfall 2).
    - Parse stdout via `parseCredentialOutput`: returns `Record<string,string>` keyed by the substring before the first `=` on each non-empty line (skip lines where `indexOf("=") <= 0`).
    - If parsed `username` or `password` is `undefined`, return `null`.
    - Else return `{ username, password }` (NO `headers`, NO `cancel` -- the Phase 30 GitCredentials shape allows omitting both because they're optional).

    `credentialApprove(host, cred)` behavior (AUTH-06):
    - Builds input via `buildAttributeBlock(host, cred)` -- output: `"protocol=https\nhost=<host>\nusername=<u>\npassword=<p>\n\n"`. Username/password emitted ONLY when defined on the cred (RESEARCH Example 1 Pattern 3 supports the optional-field shape).
    - Wraps `gitCredentialIO("approve", input)` in try/catch; on any failure (subprocess error, non-zero exit) silently return (RESEARCH Pattern 3 + CP-7 -- persistence is best-effort; the in-memory token still works).

    `credentialReject(host, cred)` behavior (AUTH-07 prereq for Phase 32):
    - Same shape as approve; spawns `"reject"` subcommand. Same best-effort silent-return on failure.

    Internal helper `buildAttributeBlock(host: string, cred?: GitCredentials): string`:
    - Always emits `protocol=https` and `host=<host>` first.
    - When cred is supplied, additionally emits `username=<u>` (only if cred.username !== undefined) and `password=<p>` (only if cred.password !== undefined).
    - Joins with `\n` and appends `\n\n` (the blank-line terminator per git-credential wire format).
    - NEVER emits a `path=` line (RESEARCH Pitfall 4).

    Internal helper `parseCredentialOutput(stdout: string): Record<string,string>`:
    - Splits stdout on `\n`, skips lines where `indexOf("=") <= 0`, builds the dict per RESEARCH Pattern 2.
