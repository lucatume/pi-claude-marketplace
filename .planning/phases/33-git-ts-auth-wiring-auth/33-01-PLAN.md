<objective>
Phase 33 ships the isomorphic-git auth-callback assembly inside
`platform/git.ts`: a new `buildAuthCallbacks` factory that takes
`CredentialOps` (Phase 31), a `host`, and a caller-provided
`onAuthRequired` handler, and returns the `{ onAuth, onAuthFailure }` pair
that isomorphic-git's `clone`/`fetch` consume.

Behavior (per AUTH-01 / AUTH-02 / CP-9 / CP-10):

1. `onAuth` consults `credentialOps.fill(host)` first. On a hit it returns
   the stored credential immediately (AUTH-02 silent reuse). On a miss it
   invokes `opts.onAuthRequired()`; on `{ ok: true, cred }` it sets a
   closure-scoped `deviceFlowAttempted = true` and returns the new cred;
   on `{ ok: false }` it returns `{ cancel: true }`. Any exception thrown
   by `fill` or `onAuthRequired` is caught and turned into `{ cancel: true }`
   (CP-10).
2. `onAuthFailure` always calls `credentialOps.reject(host, cred)` (inside
   try/catch) and returns `{ cancel: true }`. The `deviceFlowAttempted`
   closure flag is the CP-9 retry-loop guard: after Device Flow ran once,
   we never retry; the stale-keychain case (no Device Flow yet) also
   returns cancel because isomorphic-git will then invoke `onAuth` again
   on the next operation, which falls through to Device Flow on fill miss.

Why local types instead of importing from `domain/`:
`platform/` may import from `shared/` only (`platform/README.md`).
`domain/github-auth.ts` is OFF LIMITS to `platform/`. We therefore declare
a LOCAL `AuthAttemptResult` discriminated union and an `OnAuthRequiredFn`
type in `platform/git.ts`. The shapes are structurally identical to the
`DeviceFlowResult` exported by `domain/github-auth.ts` so Phase 35
orchestrators bridge them by passing `initiateDeviceFlow` directly --
TypeScript's structural typing accepts the assignment with no adapter.

`CloneOptions` and `FetchOptions` gain a single optional `auth?:
{ credentialOps; host; onAuthRequired }` bundle. When `opts.auth` is
provided, `clone()`/`fetch()` build the callbacks and pass them through to
isomorphic-git. When omitted, behavior is identical to today's public-only
path (NFR-5 surfaces untouched).

Phase 33 is INTENTIONALLY upstream of `GitOps` (`orchestrators/marketplace/shared.ts`)
and of every clone/fetch call site. Phase 34 threads `onAuthRequired`
through `GitOps` / `DEFAULT_GIT_OPS` / `refreshGitHubClone`; Phase 35
wires the orchestrators. Phase 33's `npm run check` GREEN gate proves
the new code compiles and behaves correctly in isolation.

Purpose: make Phase 34's threading work mechanical (the interface ledge
to attach `onAuthRequired` to already exists) and lock CP-9 / CP-10 in
unit tests before any cross-tier wiring lands.

Output: `platform/git.ts` extended with the auth surface; one new test
file `tests/platform/git-auth-callbacks.test.ts`; `npm run check` GREEN.
</objective>

      The factory body owns a `let deviceFlowAttempted = false;` closure flag.

      `onAuth(url)` behavior:
        1. Try block scope start.
        2. `const filled = await opts.credentialOps.fill(opts.host);`
        3. If `filled !== null`: return `filled` (AUTH-02 silent reuse). Do NOT set `deviceFlowAttempted` to true here -- the fill hit means we have not invoked Device Flow yet.
        4. If `filled === null`: `const result = await opts.onAuthRequired();`

            - If `result.ok === false`: return `{ cancel: true }`. (Setting `deviceFlowAttempted = true` in the !ok branch is OPTIONAL; CP-9's correctness does not depend on it because a subsequent fill miss would also fail.)
        5. Catch any thrown error and return `{ cancel: true }` (CP-10 -- no raw propagation to isomorphic-git).

      `onAuthFailure(url, cred)` behavior:
        1. Try block scope start.
        2. `await opts.credentialOps.reject(opts.host, cred);` -- evict the stale credential.
        3. Return `{ cancel: true }` (CP-9 -- never retry from this seam; isomorphic-git's retry loop is broken here).
        4. Catch any thrown error and STILL return `{ cancel: true }` (CP-10).

      Note: the `deviceFlowAttempted` flag is read by `onAuthFailure` ONLY for clarity / future-proofing. The current implementation returns `{ cancel: true }` in all paths inside onAuthFailure regardless of the flag (because retrying Device Flow inline would itself loop). The flag is set in `onAuth` so a future enhancement could differentiate stale-keychain rejection from post-DF rejection; for SC-2 it is sufficient that onAuthFailure always returns cancel.
