<objective>
Wire the load-time backfill into `applyReconcile`: gate the scan on the
`lastReconciledExtensionVersion` stamp (the only input that can move the supported-kind
boundary), re-materialize each force-installed plugin whose supported set grew via the now
force-capable reinstall primitive (cache-only, NFR-5), fold promotion rows into the single
reconcile cascade (RECON-04), and stamp the running version whenever the gate opened. This
is the integration plan that ties together the version constant + stamp field (68-01), the
force-capable reinstall path (68-02), and the backfill outcome/projection (68-03) into the
BFILL-01/BFILL-02 behavior.

The step runs in `applyReconcile`'s per-scope apply region with NO outer lock — `reinstallPlugin`
self-locks per scope and `proper-lockfile` is not re-entrant (CR-01); it must not run inside
the write-free read pass. The stamp is written via `withStateGuard` -> `saveState` (SPLIT-02 /
NFR-1), never a bare `atomicWriteJson`. Severity on promotion rows is a sensible default
(info, from 68-03) — Phase 69 finalizes it; Phase 70 freezes the byte-exact token.

Purpose: Deliver the user-observable behavior — a force-installed plugin's previously-skipped
components re-materialize automatically at load once the extension supports them, with no
manual command and no upgrade.
Output: `applyBackfillForScope` + its wiring + the integration test file.
</objective>
