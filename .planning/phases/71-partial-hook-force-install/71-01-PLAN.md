<objective>
Convert the hooks supportability gate from reject-all to an accumulating PARTITION. `checkMatcherSupportability` (currently first-fail) becomes `partitionHooks`, returning the supported `HooksConfig` strict subset plus a `dropped` enumeration of every skipped event / matcher-group / handler. `parseHooksConfig`'s success arm returns the filtered subset as `value` and threads `dropped`; the pre-existing structural arms (invalid JSON S1, schema failure S2) stay untouched so structural precedence is preserved (PHOOK-03). Add the synthetic partial-hook fixtures and migrate the unit + architecture tests.

Purpose: Foundation primitive for the whole phase. Once `parseHooksConfig.value` is the filtered subset and `dropped` is available, the resolver (Plan 02) can route degradable drops to `partial.unsupported`, and the materialize sites stage the subset by construction (Plan 04). This is also where Open Question Q1 is locked.

Output: `DroppedHook`/`HooksPartition` types + `partitionHooks` in `hooks.ts`; widened `HookConfigParseResult`; three fixtures; migrated `hooks.test.ts` and `hooks-supportability.test.ts`.

Locked decisions implemented here:
- D-71-01: partition at BOTH event and matcher-group granularity.
- D-71-02: mixed event installs only the supportable groups.
- D-71-03 (parse half): S1/S2 stay structural; only supportability becomes degradable.
- Q1 (locked as a D-71 sub-decision): a non-command handler (P6) drops at HANDLER granularity â€” filter the group's `hooks` array; if the group empties, drop the group; if the event empties, drop the event. This maximizes installed surface and satisfies PHOOK-04 "never stage a dropped handler".
