# Phase 65.1 Plan 01: Pre-Retirement Will-Grammar Anchors Summary

Landed the two additive, green-on-arrival artifacts that anchor the will-grammar
retirement before it happens: a WILL-02 agreement test asserting the pending
`will`-grammar matches the reload-hint oracle, and the
`marketplace-remove-with-installed-plugins` catalog-uat byte state (the WILL-03
reload-deferred cascade target shape).

## What Was Built

### Task 1 -- marketplace-remove cascade catalog-uat byte state

Added a new `/claude:plugin pending` catalog-uat state keyed
`marketplace-remove-with-installed-plugins`, structurally copied from the
surviving `plugin-pending-uninstall` fixture: a status-less (bare list-arm
header) marketplace block whose `plugins` array carries two
`{ status: "will uninstall", name }` child rows (`p1`, `p2`) under scope `user`.
Byte form (confirmed by the byte-equality runner):

```text
● old-mp [user]
  ○ p1 (will uninstall)
  ○ p2 (will uninstall)
```

This is the WILL-03 / D-65.1-03 shape: marketplace de-registration is immediate
(no `will remove` token); only the plugin-uninstall cascade is reload-deferred,
surfaced as per-plugin `will uninstall` rows. No new status token and no new
render arm were invented -- it reuses the existing bare list-arm header +
`renderWillUninstall` row form. The `mp-add-plugin-install` fixture was left
unchanged (still `status: "will add"`), as instructed.

### Task 2 -- WILL-02 reload-hint agreement test (new isolated file)

Created `tests/architecture/notify-will-reload-agreement.test.ts`. It pins the
WILL-02 invariant: a pending row carries `will` exactly when its REALIZED command
cascade emits the `/reload to pick up changes` trailer. Two tests:

- Reload-deferred realized cascades (plugin install, uninstall, enable, disable,
  and the marketplace-remove plugin-uninstall child cascade) MUST emit the
  trailer -- so their pending rows correctly keep a `will` token.
- The retired marketplace-level actions (realized `(added)` cascade, realized
  header-only `(removed)` cascade with no installed plugins) MUST NOT emit the
  trailer -- so their pending tokens are correctly dropped.

The oracle (`shouldEmitReloadHint`) and trailer literal (`RELOAD_HINT_TRAILER`)
are module-private in `shared/notify.ts`; the test observes the oracle through
its only public effect -- the trailer's presence in the rendered `notify()`
output -- mirroring the seam `notify-grammar-invariant.test.ts` already uses.
