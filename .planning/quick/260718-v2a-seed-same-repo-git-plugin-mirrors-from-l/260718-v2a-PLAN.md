<objective>
Close the same-repo gap in the v0.9.0 plugin clone-cache model: when a marketplace
manifest declares a git-source plugin (url / github / git-subdir) whose canonical
clone URL is the same repository the marketplace itself lives in, seed the plugin's
unpinned mirror (`plugin-clones/<urlhash12>/`) — or its per-sha pinned clone when the
pin is reachable — from the local marketplace checkout at `marketplace add` time,
instead of re-cloning identical bytes over the network. After seeding, the existing
read-only presence probe finds the warm mirror on its own, so the plugin stops
rendering `(remote)` right after `add`.

Purpose: eliminate redundant network on same-repo git plugin sources and the
misleading `(remote)` status they show after `marketplace add`. This is robustness /
UX polish (no official marketplace ships same-repo git sources today — they use
`path`), so the effort is sized as a single additive seam plus a write-path hook.

Output: a `seedSameRepoPluginMirrors` entrypoint in the git-surface-allowed
`clone-cache.ts` seam, a best-effort post-commit call from `marketplace add`, and
behavioral tests for every acceptance bullet. No changes to the presence probe's
read-only contract, no changes to GC.
</objective>

<decisions>
The two design decisions the task requires be made in-plan (not left to the executor)
are locked here. Cite these IDs in source comments (decision IDs are allowed by
`.claude/rules/typescript-comments.md`; phase/plan/wave numbers are NOT).

D-SEED-01 — SEEDING MOMENT = marketplace add, post-commit, best-effort.
  Rationale: the acceptance criteria require the plugin to render non-`(remote)`
  immediately after `marketplace add` with zero additional network. The presence
  probe is a READ surface — it must not mutate — so it only reports a warm mirror if
  `plugin-clones/<key>/` already exists on disk. Therefore the mirror must be created
  on the add WRITE path. Seeding at first fetch/install would be too late for the
  status-after-add criterion (and fetch.ts is forbidden the git surface by the
  no-orchestrator-network gate, forcing all-seam indirection). The seeding call runs
  AFTER `tx.save()` succeeds, in the same swallowing try/catch tier as the existing
  completion-cache invalidation, so a seeding failure can never roll back the
  already-committed add (NFR-3 retry-safety). install / fetch continue to handle a
  cold mirror through their normal materialize path, so no second hook is needed.

D-SEED-02 — CASE-B GIT-METADATA READ = yes, at add time, network-free, silent-degrade.
  Reading a local checkout's origin remote is a LOCAL read of the checkout's own
  `.git` metadata — no network — so it does NOT violate NFR-5's "path-source add must
  not touch the network". It is read-only against the marketplace root's `.git`
  (never mutates the user's checkout), it is the ONLY way to discover a path-source
  marketplace's canonical URL, and it is bounded to the add write-path (never a read
  surface). Mechanism: read `<marketplaceRoot>/.git/config` fs-only (parse the
  `[remote "origin"]` `url`), mirroring the existing fs-only `.git`-reader
  `readMirrorHeadSha` in git-source-probe.ts — NO `git` subprocess, NO new GitOps
  primitive. The discovered origin string is run through the EXISTING
  `parsePluginSource` + `canonicalCloneUrl` (no second canonicalization) and compared
  against each plugin source's canonical URL. A miss — no `.git/config`, no origin
  remote, a non-git directory, or an SSH-form origin that does not canonicalize to the
  plugin's https URL — simply does not seed; the plugin keeps its normal `(remote)` →
  network-clone behavior. Seeding is best-effort UX polish, never an error path.

D-SEED-03 — SEED MECHANISM = tree copy + atomic rename; origin preserved by
  construction; pinned gated by a checkout attempt.
  Seed by copying the marketplace checkout's tree (`cp` recursive from
  node:fs/promises — cross-FS safe, already used in bridges/skills/stage.ts) into a
  fresh `sources-staging/<uuid>/`, then an atomic SAME-FS rename into
  `plugin-clones/<key>/` (reuses the clone-cache staging machinery; NFR-1). Copying
  the checkout's `.git` preserves the correct origin BY CONSTRUCTION: Case A's
  `sources/<mp>/` was cloned FROM the canonical URL, and Case B's URL was READ FROM
  that very origin — so in both cases the seeded mirror's origin already equals the
  real remote URL (asserted by test, D-SEED-05 contract). A local-path clone is NOT
  used precisely because it would set origin to the local path. Unpinned mirrors track
  the copied HEAD (no checkout). For a pinned source, `gitOps.checkout({dir, ref: sha})`
  the staged copy: a successful checkout IS the reachability proof (seed it); a
  `CommitNotFetchedError` (or any throw) means the pin is not in the local history —
  clean the staging dir and skip, so the normal network path materializes it later.
</decisions>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| marketplace checkout → plugin-clone cache | A tree (working files + `.git`) from `sources/<mp>/` (Case A, cloned by the extension) or an arbitrary user path (Case B) is copied into `plugin-clones/<key>/`. |
| `.git/config` origin string → canonical-URL match gate | An origin URL read from the checkout's git metadata drives the same-repo match decision. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-SEED-01 | Tampering | seed destination path (`plugin-clones/<key>/`) | high | mitigate | Destination is composed ONLY from `pluginMirrorKey`/`pluginCloneKey` outputs and routed through `locations.pluginCloneDir(key)` — the SC-7/NFR-10 containment chokepoint (assertSafeName + assertPathInside). No manifest/user string joins the path. |
| T-SEED-02 | Information disclosure / NFR-5 | Case-B origin read | medium | mitigate | Origin is read fs-only from `<marketplaceRoot>/.git/config`; NO `git` subprocess, NO network — a local metadata read on a checkout the user already added. Consumed only on the add write-path, never on a read surface. |
| T-SEED-03 | Tampering | non-atomic clone commit | medium | mitigate | Copy lands in `sources-staging/<uuid>/`; the visible `plugin-clones/<key>/` appears only via an atomic same-FS rename (NFR-1); EEXIST/ENOTEMPTY treated as a concurrent-winner warm cache. |
| T-SEED-04 | Spoofing | wrong-content per-sha entry | high | mitigate | A pinned source is seeded ONLY after `gitOps.checkout(sha)` succeeds against the copied history; an unreachable pin is skipped (network fallback), so an immutable per-sha entry is never fabricated from non-matching content. |
| T-SEED-05 | Denial of service | seeding failure blocking add | low | accept | Seeding runs post-commit inside a swallowing try/catch and per-entry try/catch; any failure leaves the plugin `(remote)` for the normal network path and never disturbs the committed add (NFR-3). |
</threat_model>
