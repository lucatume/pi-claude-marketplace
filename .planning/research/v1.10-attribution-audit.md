# v1.10 Research -- Error Attribution & Message-Type Audit

**Produced:** 2026-06-07, by a three-part parallel codebase audit (message/type-system
consistency; plugin-op attribution; marketplace-op attribution). Read-only investigation.
This is the research input for milestone v1.10 (formal 4-agent domain research was skipped --
this audit serves that role for an internal correctness/refactor milestone).

**Origin:** Two `.planning/BACKLOG.md` items (install marketplace-missing misattribution;
structural `{not added}` `PluginInfoMessage` variant) were generalized into a full sweep of
every plugin and marketplace operation.

The closed `Reason` set lives at `shared/notify.ts` (the `REASONS` tuple). It has **no
`marketplace not added` member today**; `info` reuses `not added` for the marketplace-missing
row. **v1.10 decision: reuse `not added` as the single canonical reason** for "marketplace not
present in scope" (no new REASONS member; every op converges on `info`'s model).

---

## Bug classes

- **Class A -- Attribution errors:** a precondition failure (most often a missing/not-added
  marketplace) is blamed on the wrong subject (the plugin) or with a wrong/lying reason.
- **Class B -- Type foot-guns:** runtime-only invariants, placeholder fields, and illegal-but-
  typeable shapes the type system permits.
- **Class C -- Cross-op inconsistency:** the same precondition reported differently (or thrown
  raw) by different operations.
- **Class D -- Network-policy:** a no-network op (NFR-5) reports a network failure.

---

## The throughline

`info` (the newest op, v1.8) attributes the marketplace-missing condition correctly:
`(failed) {not added}` on the **marketplace** subject row. Every older op gets it wrong --
blames the plugin, goes silent, or throws raw past the notify boundary -- and the type model
permits all of them to stay wrong. v1.10 makes every op converge on `info`'s model and makes
the drift-permitting shapes unrepresentable.

---

## Theme 1 -- "Marketplace not added" is misattributed in every op except `info`

Identical precondition (target marketplace absent in the requested scope), six behaviors:

| Op | Current behavior | Verdict |
|----|------------------|---------|
| `info` | `(failed) {not added}` on the **marketplace** row | correct (the model to copy) |
| `install` | `(failed) {not in manifest}` on the **plugin** row | wrong subject + wrong reason (backlog #1) |
| `update` | `(failed) {not found}` on plugin row / `(<mp>)` synthetic row | wrong subject |
| `reinstall` | `(skipped) {not installed}` (explicit scope) *or* `(failed) {not found}` (no scope) | wrong + self-inconsistent |
| `uninstall` | **silent -- no output** | hides a typo'd marketplace |
| `remove` / `update` | **raw `MarketplaceNotFoundError` thrown past the orchestrator** | bypasses notify entirely |

No canonical reason: install reuses `not in manifest`, info uses `not added`, autoupdate uses
`not found` -- three vocabularies for one condition.

### Plugin-op precondition matrix (key rows)

Cell = reason token + subject. `MISATTRIB` = wrong subject/reason. `--` = not reachable.

| Condition | install | uninstall | reinstall | update | info | list |
|---|---|---|---|---|---|---|
| **Marketplace absent (target scope)** | `{not in manifest}` plugin-row **MISATTRIB** (A-1) | **silent** **MISATTRIB** (A-5) | `{not installed}`/`{not found}` **MISATTRIB** (A-4) | `{not found}` plugin/`(<mp>)` row **MISATTRIB** (A-3) | `{not added}` marketplace-row **correct** | absent (ok-ish) |
| **Marketplace in OTHER scope only** | `{not in manifest}` **MISATTRIB** (A-7) | silent **MISATTRIB** (A-7) | `{not installed}`/`{not found}` **MISATTRIB** (A-7) | `{not found}` **MISATTRIB** (A-7) | no "other scope" hint | shown under its scope (ok) |
| Plugin not in present manifest, not installed | `{not in manifest}` correct | n/a | `{not in manifest}` correct | `{not in manifest}` correct | `{not in manifest}` correct | correct |
| Cascade/foreign-content failure | n/a | `{not in manifest}` **MISATTRIB** (A-9) | `{rollback partial}`/`{not in manifest}` fallback partial **MISATTRIB** (A-10) | `{rollback partial}` correct | n/a | n/a |
| Permission / source-missing / version | uniformly **correct** across all ops | | | | | |

The misattribution cluster is the top two rows (marketplace-existence + scope). Everything
below (permission/source/version/installability) is already correct -- the bug is specific to
**marketplace existence / scope** preconditions, plus two cascade-fallback reason degradations.

## Theme 2 -- Raw throws escape the notify boundary

`remove`, `update` (single-name), and `add` throw on preconditions; the edge handlers have **no
try/catch** (handler registered bare at `edge/register.ts`), so the error propagates raw to the
Pi command runner -- unstyled, no `(failed)` row, no closed-set reason.

- `remove.ts` / `update.ts`: `throw new MarketplaceNotFoundError(...)` for a not-added marketplace (M-1).
- `add.ts`: `throw` on every failure -- duplicate name, stale clone, unsupported source,
  path-source ENOENT, invalid manifest, clone/network -- yet `REASONS` defines
  `duplicate name` / `stale clone` / `unsupported source` / `source missing` / `invalid manifest`
  that `add` never routes through `notify(...)` (M-7).

---

## Theme 3 -- Fallback reasons lie

Unclassifiable failures degrade to a reason that asserts something false:

- `{not in manifest}` for foreign-content / IO cleanup failures: `uninstall.ts` `narrowCascadeFailure`
  default (A-9), `reinstall.ts` `narrowReason` fallback (A-10), `remove.ts` `narrowCascadeFailure`
  default + `AgentsUnstageFailureError` mapping (M-4). Implies the plugin is absent from the
  manifest when the real blocker is on-disk content / IO / permission.
- `{network unreachable}` for a **path-source** malformed/unreadable manifest in `update.ts`
  (`refreshOneMarketplace` `?? ["network unreachable"]` default; `reasonsFromCascadeError` only
  classifies `PluginShapeError` + errno, so a `SyntaxError`/schema `Error` falls through) -- a
  network reason for a no-network op = **NFR-5 violation** (M-3, Class A+D).

---

## Theme 4 -- Cross-scope blind spot

Explicit `--scope` resolution (`orchestrators/plugin/shared.ts`) short-circuits without reading
the other scope's state, so "you asked for `project` but it's in `user`" is reported as
not-in-manifest / not-installed instead of hinting the real location (A-7). Affects install,
uninstall, reinstall, update. (`info` shows the found scope but adds no "other scope" hint;
project-target install *does* fall back to user by design -- CMP-3.)

## Theme 5 -- Type foot-guns

| ID | Finding | Severity |
|----|---------|----------|
| **B-1 / M-5** | `{not added}` smuggled through `PluginInfoMessage` with placeholder `marketplaceScope: scope ?? "user"` + `marketplaceDetails: { autoupdate: false }` and a runtime-only renderer carve-out (`status==="failed" && reasons.length===1 && reasons[0]==="not added"`). **Confirmed at 3 construction sites** (`plugin/info.ts`, `marketplace/info.ts` `buildNotAddedMessage`, + the standard-path coexistence). Backlog item #2. | hi |
| **B-2** | Illegal `reasons: ["not added", "permission denied"]` is typeable; only the `length === 1` runtime guard prevents the placeholder header from co-appearing with a structural reason. `not added` is a *structural* reason sharing the open `Reason[]` field with *content* reasons. | hi |
| **B-7** | The 4-member info-`kind` set is hand-re-enumerated in `computeSeverity`, `buildSummaryLine`, `shouldEmitReloadHint`, and the `notify` early-dispatch; **3 of them lack `assertNever`**. A 6th `kind` compiles clean but silently mis-routes severity / reload-hint / summary into the cascade branch. The "add-a-5th-branch-everywhere" smell. | hi |
| **B-3** | `MarketplaceNotificationMessage` has independent optionals (`status` / `reasons` / `details`) with unconstrained co-occurrence; `reasons` is meaningful only on `skipped`, `details` only on the list surface. Silently-ignored fields. | med |
| **B-4** | Empty-array `reasons: []` sentinel as a render-time invariant (`composeReasons` returns `""` for `[]`); the type says "always has reasons." | med |
| **B-5** | `outcome.reasons[0]`-only narrowing (`marketplace/update.ts`, `reinstall.ts`) discards the rest of a typed `Reason[]`. | med |
| **B-6** | Mutable `MarketplaceBlock` accumulator (`import/execute.ts`) sets `status`/`reasons` independently across passes; final shape never type-checked as a whole. | med |
| **B-8** | `NotificationMessage` union over-breadth forces info-kind guards into cascade-only paths (amplifies B-7). | lo |

## Recommended fix direction (from the audit; not implemented)

1. Introduce a marketplace-missing subject path in install/uninstall/reinstall/update/remove/
   autoupdate mirroring `info`'s `{not added}` on the marketplace subject. Stop reusing
   `PluginShapeError{kind:"not-in-manifest"}` for the marketplace-absent case; reserve
   `not-in-manifest` for genuine plugin-absent-from-present-manifest.
2. Stop throwing raw past the orchestrator boundary in remove/update/add -- route every
   precondition failure through `notify(...)` with a closed-set reason.
3. Replace the `{not in manifest}` / `{network unreachable}` fallbacks with truthful reasons;
   give cleanup/cascade failures a non-lying reason and narrow path-source manifest failures to
   the manifest-specific reason (never a network reason -- NFR-5).
4. Make explicit-scope resolution consult the other scope so the failure can say "present in
   `<other>` scope."
5. Retire the placeholder-field + runtime-carve-out variant with a dedicated message variant;
   make structural-vs-content reason mixes unrepresentable; route all info-kind consumers
   through a single `isInfoKind` guard with `assertNever`.
