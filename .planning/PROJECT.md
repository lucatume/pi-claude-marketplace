# pi-claude-marketplace

## What This Is

`pi-claude-marketplace` is a Pi extension that gives Pi users access to Claude plugin marketplaces through a `/claude:plugin` command surface intentionally aligned with Claude Code's upstream `/plugin`. It translates Claude plugin artefacts (skills, commands, agents, MCP servers) into the equivalent Pi-native artefacts (Pi skills, Pi prompt templates, pi-subagents agents, pi-mcp-adapter MCP entries) and manages their lifecycle (install, update, uninstall, reinstall, marketplace add/remove/list, import).

Fifteen milestones have shipped: v1.0 (PRD-derived successor architecture), v1.1 (atomic plugin reinstall), v1.2 (Claude settings import), v1.3 (consistent messaging -- every user-visible output conforming to a locked style guide + per-command catalog, structurally enforced by a 34-rule ESLint drift-guard plugin and a byte-equality catalog UAT runner), v1.4 (structured notification messages -- type-driven `NotificationMessage` payload replaces v1.3's string-based notify API; v1.3 drift-guard plugin retired in favor of closed-set type encoding), v1.4.1 (post-ship UAT patches -- 8 v1.4 milestone-spanning UAT gaps closed: reload-hint discipline, plugin.json version precedence, `v#<7hex>` hash display, `lsp` grammar consistency, plus runtime reproduction of the indent / tab-completion findings), and v1.5 (notification output polish -- 8 UXG output-grammar and severity-presentation refinements: benign no-ops suppressed from `Warning:`, autoupdate marker grammar, update no-op renders `(skipped)`, `<last-updated>` timestamp dropped, summary line prepended to error/warning cascades, update of manifest-absent plugin classifies as `(failed) {not in manifest}`, version arrow symmetric `v`-prefix), v1.6 (GitHub private marketplace authentication -- on-demand Device Flow auth for private GitHub marketplace sources with OS keychain storage; no env vars required), v1.7 (transaction resilience hardening -- eight saga / two-phase-commit correctness fixes covering orphan files, ghost state records, and skipped-undo failure paths), v1.8 (plugin and marketplace info commands -- `/claude:plugin marketplace info` and `/claude:plugin info` read-only detail surfaces), v1.9 (manifest in-memory cache -- process-lifetime memoization of `marketplace.json` reads with stat-keyed invalidation, byte-identical output), v1.10 (error attribution & message-type consistency), v1.11 (notification summary-line grammar), v1.12 (marketplace and plugin config files -- declarative per-scope `claude-plugins.json` with load-time reconciliation, preview, enable/disable, and command write-back), and v1.13 (Claude Code hooks bridge -- 8 bucket-A events SessionStart/UserPromptSubmit/PreToolUse/PostToolUse/PostToolUseFailure/PreCompact/PostCompact/SessionEnd dispatching at 100% fidelity with strict-supportability resolver gate, `if` field permission-rule matcher, and `asyncRewake` registry for background-spawn hooks). The latest release is npm `0.6.0` (2026-06-19); git tags run through `v0.6.0`.

## Core Value

A Pi user can run `/claude:plugin install <plugin>@<marketplace>` and, after `/reload`, have every supported Claude plugin component appear as a working Pi-native artefact -- atomically, recoverably, and with soft-dependency degradation that never blocks the install.

## Current Milestone: force-install

**Goal:** Let a Pi user `install`/`update --force` a *partially*-supported plugin -- install the supported components, degrade the unsupported ones, never block -- built on a **derived** force-state (not a persisted flag) and the **desired-state** severity model, with consistent status, list, completion, and load-time-backfill behaviour.

**Target features:**

- **Three-way resolver state** -- replace binary `installable: true|false` with `installable` / `unsupported` / `unavailable`; structural defects (bad manifest, malformed hooks, NFR-10 path violation) take precedence and yield `unavailable` (force can never rescue them). Two narrowing gates: `requireInstallable` (default path) and `requireForceInstallable` (`--force` path). `unavailable` exposes `pluginRoot` to neither -- the "force degrades components only, never hard failures" constraint becomes type-enforced (NFR-7 refined, not weakened).
- **Force install & update** -- `install --force` / `update --force` degrade-not-block on `unsupported` plugins; `--force` on a fully-supported plugin is a no-op. No `Warning:` summary in any force path.
- **Derived force-state, status & glyph** -- `force-installed` (◉) / `force-upgradable` (●) derived from the resolver state, no persisted flag, no migration. `force-upgradable` = the upgrade would *newly* degrade a currently-clean plugin (a force-installed plugin is never force-upgradable); the same predicate drives the token, the manual-`--force` requirement, and auto-update severity.
- **List filters & completion** -- add `--unsupported`; `--installed` spans `installed` + `force-installed`; `--force` before the plugin name offers force-eligible plugins in `install`/`update` completion (`unavailable` excluded).
- **Reinstall as a repair primitive** -- drop `--force` entirely; reinstall always overwrites everything (collisions and foreign content).
- **Load-time backfill** -- re-materialize (reinstall semantics) force-installed plugins whose previously-unsupported components became supported, gated on a new `lastReconciledExtensionVersion` stamp in `state.json` so the scan fires only when the extension itself gained a new bridge. No manual `complete` command.
- **Desired-state severity** -- wire SEV-01..09 onto the caller-stamped/desired-state notification model; the unavailable-install error splits (`unsupported` → error pointing at `--force`; `unavailable` → error with no `--force` suggestion).
- **Spec/doc reconcile** -- PRD §11, `docs/output-catalog.md`, `docs/messaging-style-guide.md` updated for the reconciled token set, derived-state severity, and the force-upgradable rules.

**Key context:** Clean-room rebuild. The earlier sticky-flag attempts (a persisted `forceInstalled` flag) were built and removed -- do **not** rebuild them; force-state is derived. Supersedes the historical `force-install-requirements.md` at repo root. The notification subsystem refactor (caller-stamped severity, desired-state output; workstream `ws-notification-refactor-2026-06-25`) is the SEV foundation this milestone wires onto. **Dropped from scope:** a global always-force config default and a manual `complete` command.

## Just Shipped: v1.13 Claude Hook Bridge

**Goal:** Add a hooks component bridge alongside the existing skills/commands/agents/MCP bridges, translating Claude plugin hook declarations into Pi extension event subscriptions and shell-outs. Take hooks off the explicit Out of Scope list.

**Target features:**

- New `hooks` component type in the resolver + state schema (parallel to `skills` / `commands` / `agents` / `mcpServers`)
- Bridge dispatch core: one composite `pi.on(...)` per Pi event type, internal per-plugin routing, literal + pipe-OR matcher translation with strict Claude-form tool-name semantics (full regex deferred — 100% first-party coverage without it)
- Per-event payload translators for the **8 bucket-A direct-1:1-map events only**: `SessionStart`, `UserPromptSubmit`, `PreToolUse`, `PostToolUse`, `PostToolUseFailure`, `PreCompact`, `PostCompact`, `SessionEnd`. These are the events where dispatch fires with 100% fidelity to the Claude Code contract — no synthesis, no loss modes. Bidirectional Pi↔Claude tool-name translation table is the load-bearing primitive (Pi emits `bash`/`read`/`edit`/...; Claude expects `Bash`/`Read`/`Edit`/...).
- Strict-supportability resolver gate: plugin `(unavailable) {unsupported hooks}` if its `hooks.json` references ANY event outside the bucket-A set OR a Claude tool with no Pi analog OR a regex matcher OR a non-`command` handler type. Bucket B (FileChanged), bucket D (Stop / CwdChanged / PostToolBatch / UserPromptExpansion / StopFailure), and soft-dep conditional Subagent events all flip plugins to unavailable in v1.13 — they are v1.14+ promotion candidates
- `if` field implementation: full permission-rule-syntax glob matching with Bash subcommand parsing (compound-separator split, `$()`/backtick recursion, process-wrapper strip), per-tool argument extraction, AND-with-`matcher` composition, explicit fail-open on parse failure. Standard Claude Code field since v2.1.85; documented at `code.claude.com/docs/en/hooks-guide`
- `asyncRewake` family implementation: bridge-owned `AsyncRewakeRegistry` spawns hooks detached, watches for exit code 2, on exit 2 injects via `pi.sendMessage({customType: "claude-hook-rewake", display: false, content: stderr_or_rewakeMessage}, { deliverAs: ctx.isIdle() ? "nextTurn" : "followUp" })`. Documented standard Claude Code field since v2.1.72; Pi has the matching primitives via `sendMessage` + `CustomMessage` with `display: false` (matches `<system-reminder>` user-invisible / model-visible semantic)
- Lifecycle integration: install / uninstall / enable / disable reconcile hooks through the v1.12 config + reconcile planner; per-plugin `hooks/` subtree containment (NFR-10 extension)
- Hook-payload-extension tolerance: schema `additionalProperties: true` at every nesting level; known additive-only extension set `{statusMessage, once, async, shell, args}` honored or silently dropped; unknown fields debug-log only

**Out of scope for v1.13:**

Four distinct categories of unsupported Claude hook events. All cause plugin `(unavailable) {unsupported hooks}` when referenced by a plugin's `hooks.json`:

- **Deferred to v1.14+ for engineering reasons (8 events):**

  These events ARE technically dispatchable but with documented fidelity gaps that conflict with the v1.13 strict-supportability stance. Each gets its own promotion REQ when v1.14+ adds it:

  - `FileChanged` (bucket B) — needs `chokidar@^5` + cross-platform CI; deferred per `PAYL-V2-01`
  - `CwdChanged` (bucket D) — bash-only filter + per-tool-call vs per-change discrepancy; deferred per `PAYL-V2-02`
  - `PostToolBatch` (bucket D) — idle-window accumulation may miss deeply pipelined runs; deferred per `PAYL-V2-03`
  - `Stop` (bucket D) — load-bearing canary for `ralph-wiggum`, `hookify`, `security-guidance`; needs idempotency guard + N-loop cap; deferred per `PAYL-V2-04`
  - `UserPromptExpansion` (bucket D) — false-positive risk on system messages; deferred per `PAYL-V2-05`
  - `StopFailure` (bucket D) — depends on Pi `agent_error` shape stability; deferred per `PAYL-V2-06`
  - `SubagentStart` / `SubagentStop` (soft-dep conditional) — async runs reliable, sync runs synthesize from `tool_call`/`tool_result` filtered on `toolName === "subagent"`; deferred per `PAYL-V2-07`

- **Blocked on upstream PRs not committed by v1.13 (9 events; documented as known limitations):**

  These need somebody to land a PR (us, upstream, or a third party) before the bridge can observe them. v1.13 does NOT depend on any of these PRs and does NOT commit to sending them.

  - 4 events needing `pi-coding-agent` runtime exposures: `Notification`, `PermissionRequest`, `PermissionDenied`, `MessageDisplay` (display-only semantic) — `EPROM-01`
  - 1 event needing a new Pi feature: `TeammateIdle` (no agent-team primitive in Pi or pi-subagents) — `FPROM-01`
  - 2 events needing two specific PRs to `pi-mcp-adapter`: `Elicitation`, `ElicitationResult` (register `ElicitRequestSchema` handler + publish on `pi.events`) — `GPROM-01`
  - 2 events needing one PR to `@zenobius/pi-worktrees`: `WorktreeCreate`, `WorktreeRemove` (publish `worktree:created` / `worktree:removed` on `pi.events` from the existing command handlers) — `GPROM-01`

- **Semantically inapplicable (5 events; permanently unsupportable, no upstream PR helps) — `HPROM-01` confirms permanent gate:**
  - `ConfigChange` — matcher values (`user_settings` / `policy_settings` / etc.) are Claude paths and concepts that don't exist under Pi
  - `Setup` — no Pi equivalent for Claude's `--init-only` CLI mode; the session model has no one-shot init phase
  - `InstructionsLoaded` — fires for `CLAUDE.md` / `.claude/rules/*.md`; Pi reads `AGENTS.md` and a different context-file model
  - `TaskCreated` / `TaskCompleted` — Pi has no canonical task primitive; rpiv-todo and pi-crew are competing extension takes with different semantics; Claude's TaskCreate sub-task-isolation model doesn't map to either

- **Other v1.13 boundaries:**
  - Full regex matchers (literal + pipe-OR covers 100% of first-party plugins) — `MATCH-V2-01`
  - Non-`command` handler types (`http` / `mcp_tool` / `prompt` / `agent`) — `TYPES-V2-01`
  - Claude tools with no Pi analog (`MultiEdit`, `NotebookEdit`, `NotebookRead`, `WebFetch`, `WebSearch`, `Task`, `TodoWrite`, `TodoRead`, `ExitPlanMode`, ...) — `PROM-01`

**Marketplace coverage under v1.13 strict bucket-A scope:** 10/13 first-party plugins install (76.9%); 2/5 hook-using plugins (40%). `ralph-wiggum`, `hookify`, and `security-guidance` flip to UNAVAILABLE due to using `Stop` (bucket D); `security-guidance` is additionally blocked by `MultiEdit`/`NotebookEdit` tool-name gap. v1.14+ promotion of `PAYL-V2-04` (Stop synthesis with engineered safeguards) restores `ralph-wiggum` + `hookify`; `security-guidance` additionally needs `PROM-01` (tool-name mapping).

## Constraints

- **Runtime:** Node >= 20.19.0 (NFR-4)
- **Tech stack:** TypeScript strict; the resolver MUST expose discriminated `installable: true | false` so consumers cannot read `pluginRoot` from a non-installable plugin (NFR-7)
- **Pi API:** `@mariozechner/pi-coding-agent` peer dependency, currently `*` with development against `^0.70.6`; pinning a min version is a successor SHOULD (NFR-11)
- **File operations:** All disk mutations atomic (tmp + rename or atomic JSON write) -- NFR-1
- **Recovery model:** No fix may require a Pi process restart; `Run /reload` must suffice (NFR-2). All operations must be safe to retry -- idempotent or fail-clean (NFR-3)
- **Network policy:** Network is required only for GitHub-source `marketplace add` and for `update`/`marketplace update` against GitHub-source marketplaces; `install`, `list`, `uninstall`, `marketplace remove`, and path-source `marketplace add` MUST NOT touch the network (NFR-5)
- **Containment:** Refuse to write outside `<scopeRoot>/pi-claude-marketplace/`, `<scopeRoot>/agents/`, or `<scopeRoot>/mcp.json` (NFR-10)
- **Quality bar:** `npm run check` must stay green -- typecheck + ESLint + Prettier + tests (NFR-6)
- **Output channel:** All user-visible messages MUST go through `ctx.ui.notify(message, severity)`; direct `process.stdout`/`process.stderr` writes forbidden in command/bridge code (IL-2). Single sanctioned `console.warn` is the load-time legacy migration save failure (IL-3)
- **No telemetry V1:** No metrics, no event sink, no analytics endpoint (IL-4)
- **English only V1:** No message catalog, no locale negotiation (IL-1)
- **Scope model:** Exactly two scopes -- `user` (`~/.pi/agent/`) and `project` (`<cwd>/.pi/`). Claude Code's `local` scope is not introduced (SC-1). Marketplace records and plugin install records are scoped independently per D-29 / CMP-1..8.
