## Full hands-on sweep -- 2026-05-30 (COMPLETE)

The deferred tests (11-27) and the v0.1.7-invalid re-runs (1-10) were
executed hands-on against the v0.2.0 source via `scripts/pi.sh`
source-load (no npm publish/link required -- the Phase 25 runtime path).
Result: the v2 byte-contract in `docs/output-catalog.md` PASSES end-to-end
across list / install / update / reinstall / uninstall / marketplace
add-remove-update / autoupdate / usage-errors / soft-dep, including the
v1.4.1 fixes (reload-hint discipline G-MIL-01/02/06 + G-21-01, SemVer
version display G-MIL-05/08, soft-degrade T26). Zero byte-contract
regressions.

Six product/UX findings surfaced -- all change-requests against the
contract or doc fixes, none byte-contract bugs. Disposition: PR #22 ships
the verified v2 as-is; the findings are recorded in `.planning/BACKLOG.md`
(`## v1.4 UAT findings`) for a follow-up UX milestone:

1. Drop the `<last-updated <iso>>` marker from `marketplace list` (noise;
   meaningless for path sources).
2. A benign `{up-to-date}` skip should not route at `warning` severity --
   "already up to date" is a no-op, not a warning.
3. Suppress the `Error:`/`Warning:` label on multi-line cascade output (it
   breaks the 0/2 indent ladder and duplicates the inline per-row status);
   keep it for single-line messages; keep the severity color throughout.
   Likely needs an upstream `@earendil-works/pi-coding-agent` capability.
4. Autoupdate surface should use marker tokens `<autoupdate>` /
   `<no autoupdate>` (incl. an explicit off-marker) rather than the status
   tokens `(autoupdate enabled)` / `(autoupdate disabled)`; idempotent ->
   `<autoupdate> {already autoupdate}` / `<no autoupdate> {already no autoupdate}`.
5. `marketplace update` no-op renders `(updated)` (implies a change);
   prefer `(skipped) {up-to-date}`, mirroring the plugin-level no-op.
6. Catalog correction: the `marketplace add` github-source section wrongly
   claims github sources default autoupdate ON. Actual v2 behavior: `add`
   never enables autoupdate (opt-in via `bootstrap` or explicit
   `marketplace autoupdate`). Fix the catalog prose. Plus a doc nit: the
   catalog heading `marketplace autoupdate <enable|disable>` does not match
   the real `autoupdate` / `noautoupdate` verbs.
